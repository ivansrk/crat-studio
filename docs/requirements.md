# Требования (EARS)

Формат EARS:
- Постоянные: «Система должна …»
- Событийные: «Когда <триггер>, система должна <реакция>»
- Состояния: «Пока <состояние>, система должна …»
- Нежелательные ситуации: «Если <условие>, то система должна …»

ID стабильны — на них ссылаются docs/flows.md и код-ревью. Не переиспользовать ID удалённых требований.

## 0. Публичный сайт: главная и лендинг курса (SITE)

- **SITE-01.** Система должна предоставлять публичную главную страницу CRAT studio (`/`): кто мы, чем занимаемся, список курсов (в MVP — один), переход на лендинг курса. Без аутентификации.
- **SITE-02.** Система должна предоставлять публичный лендинг курса (`/ai-basics`): для кого курс, программа (4 модуля, 12 уроков — из course.yaml, не дублировать руками), результат, мини-проект и сертификат, блок «Оставить заявку» с формой регистрации (раздел 1).
- **SITE-03.** Форма заявки живёт на лендинге (якорь `/ai-basics#signup`); отдельная страница регистрации — та же форма по прямой ссылке.
- **SITE-04.** Когда авторизованный студент открывает главную или лендинг, система должна показывать кнопку «В кабинет» вместо «Оставить заявку».
- **SITE-05.** Тексты главной и лендинга — контент: правятся без программиста (markdown/словарь), финальные тексты — от Ивана.
- **SITE-06.** Главная и лендинг подчиняются общим правилам UX (крупный кегль, стиль CRAT, никакого белого фона — раздел 11).

## 1. Регистрация по ссылке (REG)

- **REG-01.** Система должна предоставлять публичную страницу регистрации без аутентификации.
- **REG-02.** Форма должна содержать поля: имя (обяз.), фамилия (обяз.), email (обяз.), телефон (опц.), telegram (опц.).
- **REG-03.** Форма должна содержать два отдельных, непредотмеченных чекбокса: согласие на обработку персональных данных (обязательное для отправки) и согласие на рассылку (опциональное).
- **REG-04.** Когда форма отправлена валидно, система должна создать запись `registrations` и записи `consents` (тип, значение, дата, источник `registration_form`) и показать страницу «Заявка принята, доступ выдаст администратор».
- **REG-05.** Когда отправлена форма с email, по которому заявка уже существует, система должна не создавать дубль, а обновить существующую заявку (поля, `submitCount+1`, статус `RESUBMITTED`) и показать ту же страницу «Заявка принята».
- **REG-06.** Если чекбокс согласия на обработку данных не отмечен, то система должна отклонить отправку с понятным сообщением и не создавать записей.
- **REG-07.** Когда с одного IP поступает 6-я попытка регистрации за час, система должна отклонить её с мягким сообщением «Слишком много попыток, попробуйте позже» (лимит 5/час/IP).
- **REG-08.** Email должен нормализоваться (trim, lowercase) до любых сравнений и записи.
- **REG-09.** Если форму отправляет email уже зачисленного студента, то система должна обновить заявку как в REG-05 и пометить её признаком «уже зачислен» для админки (доступ повторно не выдаётся автоматически).

## 2. Вход: magic link и сессии (AUTH)

- **AUTH-01.** Система должна выполнять вход студента и админа только по magic link, отправляемой на email; паролей не существует.
- **AUTH-02.** Когда пользователь запрашивает ссылку входа, система должна: если email принадлежит пользователю с доступом — создать токен и отправить письмо; в любом случае показать одинаковый ответ «Если у вас есть доступ, письмо отправлено» (защита от перебора email).
- **AUTH-03.** Токен должен: жить 15 минут; быть строго одноразовым; храниться в базе только в виде SHA-256-хэша (D-009).
- **AUTH-04.** Когда пользователь переходит по действующей ссылке, система должна атомарно пометить токен использованным (`UPDATE … WHERE usedAt IS NULL`), создать сессию и перенаправить в кабинет (админа — в админку).
- **AUTH-05.** Если ссылка уже использована, то система должна показать страницу «Ссылка использована, запросите новую» с кнопкой запроса.
- **AUTH-06.** Если срок ссылки истёк, то система должна показать страницу «Ссылка устарела, запросите новую» с кнопкой запроса.
- **AUTH-07.** Сессия должна жить 30 дней и храниться в httpOnly, Secure, SameSite=Lax cookie, подписанной `SESSION_SECRET` (stateless, D-008).
- **AUTH-08.** Когда с одного email запрошена 4-я ссылка за 15 минут, система должна отклонить запрос с мягким сообщением (лимит 3/15 мин/email).
- **AUTH-09.** Роль ADMIN должна определяться вхождением email в `ADMIN_EMAILS`; при старте приложения система должна создать/синхронизировать админ-пользователей (seed).
- **AUTH-10.** Если email пользователя с активной сессией удалён из `ADMIN_EMAILS`, то система должна отказывать в админ-доступе при первом же запросе (роль проверяется по env на каждый админ-запрос, не по cookie).
- **AUTH-11.** Пока пользователь не аутентифицирован, система должна перенаправлять с защищённых страниц на страницу запроса ссылки.

## 3. Админ-панель (ADM)

- **ADM-01.** Пока пользователь не админ, система должна возвращать 404/редирект для всех роутов и API админки.
- **ADM-02.** Система должна показывать список заявок (статус, даты, признак повторной отправки и «уже зачислен») и список студентов.
- **ADM-03.** Когда админ нажимает «выдать доступ», система должна: создать `User` из заявки (если нет), создать `Enrollment` (source=`manual`), отправить письмо «доступ выдан» с magic link, обновить статус заявки на `ENROLLED`.
- **ADM-04.** Если доступ уже выдан (двойной клик, два админа одновременно), то система должна опереться на уникальность `(userId, courseSlug)` и показать «доступ уже выдан», не создавая дубль и не отправляя второе письмо.
- **ADM-05.** Система должна показывать прогресс каждого студента: по урокам (квиз/практика/пройден), статус мини-проекта, отложенные вопросы, состояние сертификата.
- **ADM-06.** Когда админ проверяет мини-проект, система должна позволить выставить `approved` или `needs_changes` с обязательным комментарием при `needs_changes`.
- **ADM-07.** Если два админа одновременно редактируют один объект (проверка проекта), то система должна сравнить `updatedAt` и при конфликте показать «данные изменились, обновите страницу», не затирая чужое решение молча.
- **ADM-08.** Система должна позволять переотправить любое письмо из `email_log` вручную.
- **ADM-09.** Когда админ запрашивает экспорт CSV, система должна выгрузить только контакты с действующим согласием на рассылку (последняя запись consents типа NEWSLETTER = granted): имя, фамилия, email, телефон, telegram, дата согласия.
- **ADM-10.** Когда админ нажимает «удалить студента и все данные» и подтверждает действие, система должна выполнить GDPR-удаление: каскадно удалить пользователя, заявку, согласия, прогресс, результаты квизов, отложенные состояния, сабмишены, email_log, magic links; сертификат — обезличить и отозвать (CERT-07).
- **ADM-11.** GDPR-удаление должно требовать явного подтверждения (ввод email студента) и быть необратимым.

## 4. Кабинет студента (CAB)

- **CAB-01.** Кабинет должен показывать прогресс курса: модули как «здания с неоновыми вывесками», фигурка студента движется по горизонту соответственно доле пройденных уроков (в Ф0–Ф4 — семантическая разметка + простая визуализация, полный арт — Ф5).
- **CAB-02.** Кабинет должен показывать личную миссию студента: текстовое поле, впервые заполняется в уроке 1.1, редактируемо из кабинета в любой момент, видно на главной кабинета.
- **CAB-03.** Кабинет должен показывать список уроков со статусами: недоступен без enrollment / не начат / в процессе / пройден.
- **CAB-04.** Когда студент входит в кабинет и существуют уроки, пройденные ≥7 дней назад (по Europe/Warsaw), без сданных отложенных вопросов, система должна показать блок «3 вопроса на 2 минуты» (вычисление в момент запроса, без cron — D-005).
- **CAB-05.** Когда студент отвечает на отложенные вопросы, система должна записать результат в `deferred_quiz_state`, показать пояснения к ответам и не изменять статус «пройден» урока.
- **CAB-06.** Если отложенных блоков накопилось несколько, то система должна показывать по одному за визит — самый давний.

## 5. Страница урока (LES)

- **LES-01.** Страница урока — сверху вниз: видео (если задан `video_id` в meta.yaml — вверху) → контент урока (LES-04) → «Скачать шпаргалку» → кнопка **«Завершить урок»**. Когда студент нажимает «Завершить урок», система должна открыть квиз отдельным шагом (в Ф5 — «сцена с прожектором»); после квиза — практика → ссылка на следующий урок.
- **LES-02.** Видео должно встраиваться через Kinescope embed по `video_id` из meta.yaml; дополнительно `<Video kinescope>` может размещаться в любом месте контента урока.
- **LES-03.** Если `video_id` пуст, то система должна показать аккуратный плейсхолдер «Видео скоро появится»; остальные блоки урока работают полностью.
- **LES-04.** Контент урока должен рендериться из lesson.mdx: markdown + компоненты белого списка (Figure, Gallery, Callout, Video, Download, Trainer, Animation, Divider — docs/content-format.md §4).
- **LES-15.** Если lesson.mdx содержит компонент вне белого списка, произвольный JS/JSX или ссылку на несуществующий asset, то система должна пометить урок ошибкой валидации и показать заглушку (поведение CNT-03); «молчаливый» пропуск компонента запрещён.
- **LES-16.** Анимации в уроках — только из библиотеки платформы (`lib/design/animations`) по id; они должны уважать `prefers-reduced-motion` и не мешать чтению (аудитория без технической подготовки).
- **LES-05.** Если cheatsheet.pdf существует, то система должна показать кнопку «Скачать шпаргалку»; если нет — блок не отображается.
- **LES-06.** Пока у пользователя нет enrollment на курс, система должна не давать доступ к страницам уроков.
- **LES-07.** Квиз должен содержать 3 вопроса из quiz.yaml; после ответа на каждый вопрос сразу показывается верность и пояснение.
- **LES-08.** Когда студент завершает квиз, система должна записать попытку в `quiz_results` (ответы, счёт, время) и зачесть квиз при результате ≥2 из 3.
- **LES-09.** Система должна позволять пересдавать квиз без ограничений; каждая попытка пишется в базу; зачтённый квиз остаётся зачтённым.
- **LES-10.** Когда студент начинает квиз, система должна создать запись попытки (`startedAt`); если квиз брошен на середине, попытка остаётся незавершённой (`finishedAt` = null), не влияет на статус и при возврате начинается новая попытка с первого вопроса.
- **LES-11.** Практика должна отображаться из practice.md с чекбоксом «Сделал»; отметка пишется в `lessons_progress` с датой и может быть снята студентом (снятие не отменяет уже выданный сертификат).
- **LES-12.** Урок должен считаться пройденным тогда и только тогда, когда квиз сдан ≥2/3 И практика отмечена; это единственное определение, в т.ч. для сертификата (D-004).
- **LES-13.** Когда урок становится пройденным, система должна зафиксировать `completedAt` и создать `deferred_quiz_state` с `dueAt = completedAt + 7 дней`.
- **LES-14.** Урок 1.1 дополнительно должен содержать поле «личная миссия» (сохраняется в профиль, CAB-02).

## 6. Финальный мини-проект (PROJ)

- **PROJ-01.** Форма-отчёт должна содержать 7 полей: задача → инструмент → запрос → результат → что уточнил → как проверил → где применит.
- **PROJ-02.** Статусы: `draft → submitted → needs_changes → approved`; черновик сохраняется без отправки.
- **PROJ-03.** Когда студент отправляет отчёт, система должна перевести его в `submitted` и показать «отчёт на проверке».
- **PROJ-04.** Когда админ ставит `needs_changes` с комментарием, система должна показать студенту комментарий и открыть новую попытку (черновик, предзаполненный прошлым отчётом); число пересдач не ограничено.
- **PROJ-05.** Когда админ ставит `approved`, система должна проверить условие сертификата (CERT-01) и при его выполнении запустить выдачу.
- **PROJ-06.** Пока статус `approved`, система должна запрещать редактирование отчёта студентом.

## 7. Сертификат (CERT)

- **CERT-01.** Когда одновременно выполнено «все 12 уроков из course.yaml пройдены (LES-12)» И «мини-проект approved», система должна выдать сертификат (проверка запускается на каждом из двух событий: урок пройден / проект approved).
- **CERT-02.** Если проект не approved (включая submitted/needs_changes), то система должна не выдавать сертификат, сколько бы уроков ни было пройдено, — и наоборот.
- **CERT-03.** Номер сертификата: `CRAT-{ГОД}-{порядковый номер, 4 знака}` (год по Europe/Warsaw, счётчик в рамках года, транзакционно — без гонок и пропусков дублей). Номер уникален и никогда не переиспользуется.
- **CERT-04.** PDF (плейсхолдер-макет до Ф5): ФИО, название курса, дата, номер, бренд CRAT studio; генерация Playwright + Chromium по требованию (D-011).
- **CERT-05.** Когда сертификат выдан, система должна отправить письмо с PDF и показать копию (кнопку скачивания) в кабинете.
- **CERT-06.** Система должна предоставлять публичную страницу проверки `/cert/{номер}`: действителен → ФИО, курс, дата; не существует → «сертификат не найден».
- **CERT-07.** Когда выполняется GDPR-удаление владельца, система должна перевести сертификат в `REVOKED`, обнулить в нём ФИО и связь с пользователем; `/cert/{номер}` показывает «сертификат отозван по запросу владельца» (D-010).

## 8. Тренажёры (TRN)

- **TRN-01.** Система должна предоставлять роут `/trainers/{id}`; в MVP — страницы-заглушки для T1 «Собери запрос», T2 «Правда или нейросеть?», T3 «Опиши картинку» (спеки придут из course-factory).
- **TRN-02.** T1 должен работать через серверный прокси к Anthropic API; `ANTHROPIC_API_KEY` — только в env на сервере, в браузер не попадает никогда.
- **TRN-03.** Прокси должен ограничивать: `max_tokens` 1000; ≤20 запросов на студента в день (день по Europe/Warsaw); ≤3 запроса в минуту. Счётчики — в базе (D-015).
- **TRN-04.** Если лимит превышен, то система должна вернуть мягкое сообщение («На сегодня запросы закончились — продолжим завтра»), без технических ошибок.
- **TRN-05.** Пока пользователь не аутентифицированный студент с enrollment, прокси должен отклонять запросы (в т.ч. для тренажёров, встроенных в уроки и каталог).
- **TRN-06.** Система должна предоставлять каталог тренажёров в кабинете (`/app/trainers`) — карточки всех тренажёров с запуском; доступен только студентам с enrollment; публичной версии каталога нет; геймификации (очки, бейджи) в MVP нет.
- **TRN-07.** Тренажёр должен встраиваться в контент урока компонентом `<Trainer id>` (LES-04); встроенный тренажёр использует те же лимиты TRN-03 (общий счётчик студента, не отдельный).

## 9. Письма (MAIL)

- **MAIL-01.** Транспорт — Resend. Типы MVP: magic link, «доступ выдан», «сертификат».
- **MAIL-02.** ⚠️ Внешний блокер Ф1: домен + DKIM настраивает Иван в Resend (см. docs/phases.md, «требуется от Ивана»).
- **MAIL-03.** Каждая отправка должна логироваться в `email_log`: адресат, тип, статус (queued/sent/failed), число попыток, ошибка, id провайдера.
- **MAIL-04.** Если отправка не удалась, то система должна повторить до 3 раз с бэкоффом (1/5/15 мин, in-process — D-013); после исчерпания — статус `FAILED`, виден в админке.
- **MAIL-05.** Система должна позволять ручную переотправку любого письма из админки (ADM-08); переотправка = новая запись в email_log.
- **MAIL-06.** Маркетинговые письма обязаны содержать ссылку отписки (`/unsubscribe/{токен}`); переход пишет `consents` запись `NEWSLETTER granted=false, source=unsubscribe_link` и показывает подтверждение.
- **MAIL-07.** Транзакционные письма (magic link, «доступ выдан», сертификат) должны отправляться независимо от согласия на рассылку — отписка от рассылки не мешает учиться.

## 10. Контент (CNT)

- **CNT-01.** Контент курса — только файлы `content/{course-slug}/` по контракту docs/content-format.md; чтение — только через модуль `lib/content`.
- **CNT-02.** При старте приложения система должна валидировать весь контент; результат валидации виден в логах и в админке.
- **CNT-03.** Если yaml битый, файл отсутствует или урок из course.yaml пропал с диска, то система должна: залогировать конкретную ошибку; на странице урока показать понятную заглушку («Урок временно недоступен, мы уже чиним»); НЕ падать; существующий прогресс по уроку сохранить нетронутым.
- **CNT-04.** Для разработки в репо должны существовать заглушки всех 12 уроков `content/ai-basics/` (создаются в Ф0; реальные тексты придут из course-factory).

## 11. UX, дизайн-токены, i18n (UX)

- **UX-01.** Базовый кегль 18–20px; мелкий текст запрещён. Одна страница = одно действие.
- **UX-02.** Все строки интерфейса — в словаре `lib/i18n/` (MVP: ru); строк в разметке нет (i18n-ready).

## 12. Безопасность и NFR (SEC)

- **SEC-01.** Секретов в репозитории нет; `.env.example` содержит только имена: `DATABASE_URL, RESEND_API_KEY, ANTHROPIC_API_KEY, ADMIN_EMAILS, APP_URL, SESSION_SECRET`.
- **SEC-02.** Pre-commit с gitleaks обязателен для всех коммитов.
- **SEC-03.** Rate limits: регистрация 5/час/IP (REG-07); запрос magic link 3/15 мин/email (AUTH-08); T1 — TRN-03.
- **SEC-04.** Anthropic API вызывается только с сервера (TRN-02).
- **SEC-05.** Все мутации — только POST с проверкой сессии; admin-API дополнительно проверяет роль по env (AUTH-10).
- **SEC-06.** Ответы системы не должны раскрывать существование email в базе (AUTH-02).
- **SEC-07.** Миграции Prisma выполняются в deploy-команде Render (`prisma migrate deploy`); автобэкапы Postgres включены.
- **SEC-08.** Глубокие модули (Ousterhout): роуты тонкие, сложность прячется в `lib/*`; интерфейс модуля меньше его реализации.

## 13. Статьи (ART) — Ф6

- **ART-01.** Система должна предоставлять публичный раздел `/articles` (список) и `/articles/{slug}` (статья); статьи — файлы `content/articles/` (article.mdx, тот же движок и белый список, что уроки; docs/content-format.md §8). Публикация = git push.
- **ART-02.** Статья с `draft: true` не должна отображаться публично и попадать в sitemap.
- **ART-03.** Если статья битая (валидация), то система должна не публиковать её, залогировать ошибку и показать баннер в админке; сайт работает.
- **ART-04.** К публичному запуску раздел должен содержать 5–10 статей; статьи пишет course-factory (см. docs/phases.md Ф6).

## 14. SEO и GEO — нейропоиск (SEO) — база в Ф1, полный пакет в Ф6

- **SEO-01.** Все публичные страницы (`/`, `/ai-basics`, `/articles/*`, `/cert/{номер}`) должны рендериться на сервере (SSR/SSG) с полным HTML-контентом — без «пустых» страниц, требующих JS.
- **SEO-02.** Каждая публичная страница должна иметь уникальные title, description, canonical, Open Graph (включая OG-картинку в стиле CRAT) через Next.js Metadata API.
- **SEO-03.** Система должна отдавать `sitemap.xml` (автогенерация: публичные страницы + опубликованные статьи) и `robots.txt` (`/app`, `/admin`, служебные роуты закрыты от индексации).
- **SEO-04.** Система должна отдавать structured data schema.org (JSON-LD): Organization (главная), Course (лендинг), Article (статьи), FAQPage (при наличии FAQ-блока).
- **SEO-05.** Система должна отдавать `/llms.txt` — краткое машиночитаемое описание студии, курса и статей для нейропоисковиков (GEO); обновляется автоматически из контента.
- **SEO-06.** Публичные страницы должны укладываться в Core Web Vitals «good» (LCP < 2.5s на mobile); тяжёлые ассеты — оптимизированные изображения (next/image), ленивая загрузка ниже первого экрана.
- **SEO-07.** URL — человекочитаемые, кириллица не используется в путях; редиректы при смене URL — 301, старые URL не ломаются.
- **SEO-08.** Домен должен быть подтверждён в Google Search Console и Яндекс.Вебмастере, sitemap отправлен (действие Ивана, Ф6).

## 15. Оплата Stripe (PAY) — Ф7

- **PAY-01.** Лендинг курса должен предлагать покупку через Stripe Checkout (существующий аккаунт Ивана); цена и валюта — конфигурация, не хардкод.
- **PAY-02.** Когда Stripe присылает webhook об успешной оплате (с проверкой подписи), система должна: создать User (если нет), создать Enrollment (`source=stripe`), отправить письмо со входом; операция идемпотентна по id события Stripe.
- **PAY-03.** Ручная выдача доступа админом (ADM-03) должна продолжать работать наравне с оплатой.
- **PAY-04.** Если оплата прошла, а webhook не дошёл/упал, то админ должен видеть расхождение (страница «платежи» в админке со статусом из Stripe API) и уметь выдать доступ вручную.
- **PAY-05.** Секреты Stripe (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`) — только в env (SEC-01 дополняется в Ф7).

## 16. Google Drive (DRV) — синк в Ф6, документы в Ф8

- **DRV-01.** Исходники материалов команда ведёт в Google Drive (`edu@skld.me`); в git контент попадает скриптом синхронизации на стороне course-factory (D-023). Платформа никогда не отдаёт студентам файлы по прямым ссылкам Drive — только со своего сервера.
- **DRV-02.** (Ф8) Когда выдан сертификат или выполнен экспорт заявок, система должна складывать копию в заданную папку Drive через service account; ошибка Drive не блокирует основную операцию (лог + баннер в админке).
