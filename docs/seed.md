# scripts/seed.ts — описание

Назначение: за один запуск (`npm run seed`) привести базу к состоянию, в котором Иван может проверить текущую фазу за 10 минут, не собирая данные руками. Используется локально и на Render (после деплоя фазы).

Требования к скрипту:
- **Идемпотентен**: повторный запуск не создаёт дублей (везде upsert по естественным ключам: email, (userId, lessonId) и т.д.).
- Не трогает реальных пользователей: работает только с адресами `*@seed.crat.example`.
- Расширяется по фазам — каждая фаза добавляет свой блок, старые блоки не ломаются.

## Что создаёт (по фазам)

**Ф0:**
- Админ-пользователей из `ADMIN_EMAILS` (это же делает приложение при старте — seed проверяет/чинит).

**Ф1:**
- Заявку `zayavka@seed.crat.example` (статус NEW, оба согласия).
- Повторную заявку `povtor@seed.crat.example` (RESUBMITTED, submitCount=2, без согласия на рассылку — для проверки CSV-фильтра).
- Студента `student@seed.crat.example`: User + Enrollment + согласия (ENROLLED-заявка).

**Ф2:**
- Студенту прогресс: уроки 1.1–1.2 пройдены (квиз-попытки: одна проваленная, одна зачтённая; практика отмечена), урок 1.3 — квиз зачтён, практика нет; заполненная личная миссия; одна брошенная попытка квиза (finishedAt=null).

**Ф3:**
- Студента `diplomant@seed.crat.example` с 12/12 пройденными уроками и Submission SUBMITTED — одна кнопка approve отделяет от проверки выдачи сертификата.
- Submission NEEDS_CHANGES с комментарием у `student@seed.crat.example`.

**Ф4:**
- Бэкдейт: у `student@seed.crat.example` completedAt урока 1.1 = −8 дней, deferred dueAt = −1 день → блок «3 вопроса на 2 минуты» виден сразу.
- Записи trainer_usage на грани дневного лимита (19/20) для проверки мягкого отказа.

**Ф5:** новых данных не требует.

## Вход студентом без почты

Seed печатает в stdout готовые magic-link URL для каждого seed-пользователя (создавая токены напрямую в базе) — локальная проверка не требует работающего Resend.
