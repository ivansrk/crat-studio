# User flows, машина состояний, edge cases

Ссылки вида REG-05 — на docs/requirements.md; D-008 — на DECISIONS.md.

---

## 1. Машина состояний студента

Состояние **вычисляется из данных**, отдельной колонки нет — рассинхронизация невозможна.

```
registered → enrolled → in_progress → final_submitted → completed → certified
                              ↑              |
                              └── needs_changes
```

| Состояние | Определение (по данным) | Что видит студент |
|---|---|---|
| `registered` | есть Registration, нет Enrollment | «Заявка принята, доступ выдаст администратор» |
| `enrolled` | есть Enrollment, нет ни одной LessonProgress | кабинет, все уроки «не начат» |
| `in_progress` | есть ≥1 LessonProgress | кабинет с прогрессом |
| `final_submitted` | текущий Submission = SUBMITTED | «отчёт на проверке» |
| `completed` | 12/12 уроков completedAt И Submission APPROVED | кабинет: «сертификат готовится» (мгновенно → certified) |
| `certified` | есть Certificate VALID | сертификат в кабинете |

Переходы-нюансы:
- `needs_changes` возвращает в `in_progress`: студент дорабатывает отчёт и может параллельно дозакрывать уроки.
- Проект может быть approved раньше 12-го урока — тогда `completed/certified` наступает в момент прохождения последнего урока (CERT-01 проверяется на обоих событиях).
- Снятие чекбокса практики после выдачи сертификата состояние `certified` не отменяет (LES-11).

## 2. User flows

### F1. Регистрация
1. Гость открывает публичную страницу → форма (REG-02, REG-03).
2. Submit → валидация → rate limit 5/час/IP (REG-07).
3. Email новый → Registration NEW + 2 записи Consent (REG-04). Email существует → update, RESUBMITTED, submitCount+1 (REG-05); если уже студент — пометка alreadyEnrolled (REG-09).
4. Экран «Заявка принята, доступ выдаст администратор».

### F2. Выдача доступа (админ)
1. Админка → список заявок (ADM-02) → «выдать доступ».
2. Транзакция: User (если нет, из данных заявки; consents получают userId) → Enrollment(source=manual) → Registration.status=ENROLLED.
3. Письмо ACCESS_GRANTED с magic link → email_log (MAIL-03).
4. Повторный клик / второй админ → unique(userId, courseSlug) → «доступ уже выдан» (ADM-04).

### F3. Вход по magic link
1. Страница входа: одно поле email → rate limit 3/15мин (AUTH-08).
2. Email принадлежит User → токен (хэш в базу, TTL 15 мин) + письмо. В любом случае: «Если у вас есть доступ, письмо отправлено» (AUTH-02, SEC-06).
3. Клик по ссылке → атомарное `usedAt = now() WHERE usedAt IS NULL`:
   - успех → cookie-сессия 30 дней → кабинет (админ → админка);
   - уже использована → «Ссылка использована, запросите новую» (AUTH-05);
   - истекла → «Ссылка устарела, запросите новую» (AUTH-06).

### F4. Прохождение урока
1. Кабинет → урок: видео (или плейсхолдер LES-03) → MDX-контент урока (LES-04) → шпаргалка (если есть) → кнопка «Завершить урок» (LES-01).
2. «Завершить урок» открывает квиз отдельным шагом: старт попытки (QuizResult, startedAt) → 3 вопроса, после каждого ответа — верно/неверно + пояснение (LES-07), ответ сразу в answers → finish: score, passed = score≥2 (LES-08). Пересдача — новая попытка, безлимитно (LES-09). После квиза — практика → следующий урок.
3. Практика: чекбокс «Сделал» → practiceDoneAt (LES-11).
4. Как только quizPassedAt И practiceDoneAt → completedAt + DeferredQuizState(dueAt=+7д) (LES-12, LES-13) → проверка условия сертификата (CERT-01).
5. В уроке 1.1 — дополнительно поле «личная миссия» → User.mission (LES-14).

### F5. Отложенные вопросы (без cron)
1. Вход в кабинет → запрос: DeferredQuizState где dueAt ≤ now И answeredAt IS NULL (CAB-04).
2. Есть → блок «3 вопроса на 2 минуты» по самому давнему уроку (CAB-06). Вопросы: блок `deferred` из quiz.yaml, иначе те же 3 вопроса (D-012).
3. Ответы → answeredAt, score, пояснения. Статус «пройден» не меняется (CAB-05).

### F6. Мини-проект
1. Форма из 7 полей (PROJ-01), автосохранение черновика (DRAFT).
2. «Отправить» → SUBMITTED, submittedAt (PROJ-03).
3. Админ: APPROVED → проверка сертификата (PROJ-05); или NEEDS_CHANGES + обязательный комментарий (ADM-06) → студент видит комментарий, открывается attempt+1 DRAFT, предзаполненный прошлым отчётом (PROJ-04). Пересдачи безлимитны.

### F7. Выдача и проверка сертификата
1. Триггер: урок пройден ИЛИ проект approved → проверка «12/12 И approved» (CERT-01).
2. Транзакция: нет VALID сертификата → CertificateCounter FOR UPDATE → номер CRAT-{год}-{NNNN} (CERT-03) → Certificate.
3. PDF по требованию (D-011): письмо CERTIFICATE с вложением + кнопка в кабинете (CERT-05).
4. Публично: `/cert/{номер}` → VALID: ФИО, курс, дата · не найден: «сертификат не найден» · REVOKED: «сертификат отозван по запросу владельца» (CERT-06, CERT-07).

### F8. GDPR-удаление
1. Админка → карточка студента → «удалить студента и все данные» → подтверждение вводом email (ADM-11).
2. Транзакция: Certificate → REVOKED, fullName=null, userId=null; затем delete User (каскад сносит всё остальное) + Registration и Consent по email (ADM-10, таблица судьбы записей — в docs/data-model.md).
3. Активная cookie-сессия студента умирает сама: пользователь не находится в базе (D-008).

### F9. Отписка от рассылки
1. Маркетинговое письмо → `/unsubscribe/{токен}` (MAIL-06) → Consent(NEWSLETTER, granted=false, source=UNSUBSCRIBE_LINK) → «Вы отписаны».
2. Транзакционные письма продолжают приходить (MAIL-07) — учёбе отписка не мешает.

## Пакет Ф7 — новые флоу и edge cases (D-031…D-037, 2026-07-10)

## F10. Вход по паролю (Ф7а, заменяет F3 magic link)

1. `/login`: email + пароль → rate-limit 10/15мин/email И 20/15мин/IP (AUTH-20).
2. Пользователь найден И `passwordHash` совпал (bcryptjs.compare) → cookie-сессия 30 дней (AUTH-07) → кабинет (админ → админка). Сброс счётчика email.
3. Не найден / пароль неверен / `passwordHash=null` (не мигрированный юзер) → одинаковое «Неверный email или пароль» (AUTH-13, SEC-06). На странице всегда есть ссылка «Забыли пароль / первый вход».

## F11. Установка пароля при создании учётки (Ф7а)

Триггеры: подтверждение регистрации по инвайту (REG-13 авто) ИЛИ ручная выдача админом (ADM-03).
1. Сервер генерирует человекочитаемый пароль ≥12 симв. (D-033), пишет `passwordHash` (bcryptjs).
2. Экран успеха показывает пароль ОДИН раз + письмо `WELCOME` (пароль + ссылка на курс/вход).
3. Открытый пароль больше нигде не доступен (только хэш). Пользователь может сменить его в кабинете (F13).

## F12. Восстановление / первый вход (reset, Ф7а)

1. «Забыли пароль» (email) → rate-limit (AUTH-08) → если User есть: reset-токен (TTL 60 мин, хэш в базе, `purpose=PASSWORD_RESET`) + письмо `PASSWORD_RESET`. Ответ всегда одинаков (AUTH-16, SEC-06).
2. Клик по ссылке → атомарное `usedAt=now WHERE usedAt IS NULL`:
   - успех → форма «новый пароль» (≥8, повтор совпадает) → `passwordHash` обновлён, токен погашен → «Пароль изменён», уводим на `/login` (AUTH-17). Авто-входа нет.
   - использована → «Ссылка использована» (AUTH-05); истекла → «Ссылка устарела» (AUTH-06).
3. **Миграция существующих юзеров (D-034):** у них `passwordHash=null`. Разовая рассылка `PASSWORD_RESET` всем таким юзерам после деплоя Ф7а; плюс постоянный вход через «Забыли пароль». До установки пароля вход по F10 невозможен (AUTH-19).

## F13. Смена пароля в кабинете (Ф7а)

`/app/account`: текущий пароль + новый (≥8) + подтверждение → при верном текущем `passwordHash` обновляется; иначе ошибка без смены (AUTH-18).

## F14. Регистрация по инвайту с double opt-in → авто-доступ (Ф7б)

1. Админ создаёт инвайт (`/admin/invites`): курс, метка источника, опц. срок/лимит (INV-01). Копирует `/invite/{token}`.
2. Гость открывает `/invite/{token}`: токен активен, не просрочен, лимит не исчерпан (INV-04) → форма (имя, фамилия, email, **телефон обяз.**, telegram/whatsapp опц., чекбокс политики обяз., чекбокс рассылки — unchecked; REG-10).
3. Submit → валидация + rate-limit → `Registration` (PENDING_OPT_IN, inviteLinkId, wantsNewsletter) → токен `OPT_IN` (TTL 60 мин) → письмо `DOUBLE_OPT_IN` → «Проверьте почту» (REG-11/12).
4. Клик по ссылке подтверждения → атомарное погашение → пишутся действующие `Consent` (DATA_PROCESSING granted; NEWSLETTER granted только если чекбокс был) → **авто:** User (пароль F11), Enrollment(`source=sourceLabel`), инкремент счётчика инвайта (INV-05), Resend-синк при подписке (F17) → экран «Готово»: пароль (один раз) + ссылка на курс; письмо `WELCOME` (REG-13).

## F15. Публичная регистрация с /ai-basics с double opt-in → ручная выдача (Ф7б, D-035)

Как F14 п.2–3, но без инвайта. После подтверждения (REG-13 публичный путь): согласия фиксируются, заявка → `CONFIRMED`, экран «Заявка подтверждена, доступ выдаст администратор». Доступ выдаёт админ вручную (ADM-03 → тогда срабатывает F11: пароль + WELCOME). **Почему не авто:** курс платный, продажи ручные в MVP (D-001/D-035); авто-выдача из открытой формы раздала бы курс бесплатно.

## F16. Клиентская база в админке (Ф7б)

1. `/admin/clients`: список User (имя, фамилия, email, телефон, мессенджер, статус подписки D-014, последний курс) + поиск (CRM-02).
2. Карточка: профиль + история; редактирование имени/фамилии/телефона/telegram/whatsapp (CRM-03). Email не редактируется.
3. Кнопка «пересинхронизировать с Resend» при рассинхроне (CRM-05).

## F17. Синхронизация подписчиков с Resend Audience (Ф7б)

- Подтверждение подписки (F14/F15 REG-13) → создать/обновить контакт в Audience (`RESEND_AUDIENCE_ID`), сохранить `resendContactId` (CRM-04).
- Отписка (F9 `/unsubscribe`) → удалить/пометить контакт.
- GDPR-удаление (F8) → удалить контакт из Audience перед/после удаления User.
- Любой сбой Resend → основная операция проходит, ошибка в лог + баннер рассинхрона в админке (CRM-05). `RESEND_AUDIENCE_ID` не задан → синк выключен, платформа работает (CRM-06).

## F18. Консультация (Ф7б)

1. Форма (имя, контакт, описание; опц. направление) в кабинете `/app` и публично (`/#contact`/`/consult`) → rate-limit 5/час/IP (CONS-05).
2. Submit → `ConsultationRequest` (NEW, userId если авторизован) → письмо `CONSULTATION` админам (CONS-03) → «Заявка принята, мы свяжемся».
3. `/admin/consultations`: список, статусы NEW→CONTACTED→CLOSED (CONS-04).

## F19. Мультикурс: кабинет и маршруты (Ф7в)

1. `/app`: «Мои курсы» (Enrollment) → карточки со ссылкой на `/app/{courseSlug}`; ниже каталог остальных опубликованных курсов; неопубликованные — «Скоро» (MC-02/03).
2. Урок: `/app/{courseSlug}/lessons/{lessonId}`; проект `/app/{courseSlug}/project`; сертификат — per-course. Старые `/app/lessons/{id}` → 301 на `/app/ai-basics/lessons/{id}` (MC-04).
3. Доступ, прогресс, квизы, сертификат — по `(userId, courseSlug)`; `lessonCount(courseSlug)` — знаменатель «N/N» (MC-05/06).
