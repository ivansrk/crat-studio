### D-031. Вход по паролю вместо magic link (разворот D-003)
**Решение:** единственный способ входа студента и админа — email + пароль. Magic-link-вход отменяется (AUTH-01/02, flow F3 снимаются). Token-механика magic link (одноразовый SHA-256-хэш, TTL, атомарное гашение `usedAt`) переиспользуется как ссылка восстановления/установки пароля (TTL увеличен до 60 мин) и подтверждения регистрации (double opt-in). Модель `MagicLink` переименовывается в `PasswordResetToken` (+`purpose`).
**Причина:** прямое решение Ивана 2026-07-10. Пароль даёт мгновенный повторный вход без хождения в почту на каждый заход, привычен, работает при проблемах с доставкой писем и нужен как основа для инвайт-авто-выдачи и Stripe.
**Что теряем (честно):** (1) исчезает главное удобство magic link для аудитории 40+ — «никаких паролей»; появляются забытые пароли и обращения в поддержку (частично лечится reset-ссылкой и генерируемым паролем). (2) Пароль — новый секрет, который надо безопасно хранить (хэш) и не светить в логах/письмах (открытый пароль только в письме WELCOME один раз). (3) Нужен миграционный путь для существующих юзеров (D-034). (4) Растёт поверхность атаки (перебор пароля → AUTH-20). **Митигируется:** генерируемый сервером пароль (не слабый пользовательский), reset как «первый вход», rate-limit перебора.
**Альтернативы:** оставить magic link (отвергнуто Иваном); пароль + опциональный magic link параллельно (отвергнуто — две auth-поверхности, сложнее); OAuth-Google (отвергнуто ранее D-003, аудитория без Google-аккаунтов). Обновить CLAUDE.md (принцип «вход без паролей» → «вход по паролю»).

### D-032. Хэш пароля — bcryptjs (не argon2)
**Решение:** пароли хэшируются `bcryptjs` (чистый JS, cost-фактор ≥ 12).
**Причина:** Render собирает образ на каждый деплой; `argon2` (npm) требует нативной сборки (node-gyp/prebuilt-бинарь) — риск падения билда, особенно рядом с уже хрупким Playwright/Chromium-шагом (Ф3). `bcryptjs` — pure-JS, ноль нативных зависимостей, детерминированный деплой; CLAUDE.md уже помечает его «допустимой лёгкой» зависимостью. Наш пароль генерируется сервером (12+ симв., высокая энтропия), поэтому криптографическое превосходство argon2 против слабых пользовательских паролей здесь не решающее; bcrypt cost 12 достаточен.
**Рассмотрено:** node:crypto `scrypt` (встроен, ноль зависимостей вообще) — валидная альтернатива, но ручное управление солью/параметрами и форматом хранения; bcryptjs проще и привычнее команде. `argon2` — вернуться, если появится вход по пользовательскому паролю с высокими требованиями. [РЕШЕНИЕ АВТОРА: bcryptjs; подтвердить у Ивана либо переключить на встроенный scrypt, если хочется совсем без внешних зависимостей.]

### D-033. Сервер генерирует человекочитаемый пароль
**Решение:** при создании учётки сервер генерирует пароль ≥ 12 символов, читаемый (группы вида `xxxx-xxxx-xxxx` из набора без неоднозначных символов 0/O/1/l/I), показывает один раз на экране успеха и шлёт письмом WELCOME. Пользователь может сменить его в кабинете.
**Причина:** пользователь не придумывает пароль на регистрации (одна страница = одно действие, UX-01); генерируемый пароль сильнее и его можно сразу отправить письмом как «первый вход». Читаемый формат — чтобы можно было ввести вручную с телефона.
**Альтернативы:** пользователь задаёт пароль на форме (лишний шаг и слабые пароли); ставить пароль только через reset после первого письма (лишний цикл). Отвергнуто.

### D-034. Миграция существующих пользователей на пароли
**Решение:** `passwordHash` — nullable. Существующие юзеры (созданные до Ф7) имеют `passwordHash=null`. Путь входа: (1) постоянная ссылка «Забыли пароль / первый вход» на `/login` (AUTH-19); (2) разовая рассылка письма PASSWORD_RESET всем таким юзерам после деплоя Ф7а (идемпотентно по `passwordHash=null`). До установки пароля обычный вход невозможен, ответ — то же «неверный email или пароль» (не раскрываем состояние миграции).
**Причина:** на проде уже есть данные; нельзя разлогинить людей без пути назад. Аддитивная nullable-колонка + reset-механика (уже есть) закрывают миграцию без ручной работы.
**Альтернативы:** заставить всех пройти reset без постоянной ссылки (хрупко, если письмо потеряно); генерировать всем новые пароли и разослать (можно, но reset-ссылка безопаснее — пароль не летит в письме массово). Отвергнуто.

### D-035. Инвайт — авто-выдача; публичная заявка — ручная (двойной funnel)
**Решение:** после double opt-in регистрация ПО ИНВАЙТУ выдаёт доступ автоматически (User+пароль+Enrollment, `source=sourceLabel`); ПУБЛИЧНАЯ заявка с `/ai-basics` (без инвайта) после подтверждения email остаётся заявкой — доступ выдаёт админ вручную (ADM-03). Технически оба пути общие (double opt-in, consents, Resend-синк, письма); различие — авто-enroll vs ручной и метка источника.
**Причина:** курс платный, продажи ручные в MVP (D-001) — авто-выдача из открытой формы раздала бы курс бесплатно. Инвайт админ выдаёт осознанно уже оплатившему/приглашённому клиенту → доверие установлено, авто-выдача уместна и убирает ручное бутылочное горло для тёплого трафика.
**Альтернативы:** (а) конвертировать публичную форму в авто-выдачу тоже — отвергнуто (раздача платного курса); (б) убрать публичную форму, оставить только инвайты — отвергнуто (публичный лендинг нужен для лидов/SEO). [РЕШЕНИЕ АВТОРА — подтвердить у Ивана: публичная заявка = ручная выдача, инвайт = авто. До Stripe (Ф8) это основной способ отсекать неоплативших.]

### D-036. Мультикурс — минимально-инвазивно (схема уже courseSlug-aware)
**Решение:** курс = каталог `content/{courseSlug}/`; несколько курсов индексируются `lib/content`. Доменные функции (`lib/progress`, `lib/project`, `lib/cert`, `lib/content`) принимают `courseSlug` параметром вместо константы `COURSE='ai-basics'`. Маршруты становятся `/app/{courseSlug}/…` со 301 со старых. Единственная структурная правка схемы — `QuizResult @@unique` дополняется `courseSlug` (латентный баг: сейчас `(userId, lessonId, attempt)` без курса). `course.yaml` получает необязательный флаг `published`. Формат урока и контракт course-factory (docs/content-format.md) НЕ меняются.
**Причина:** схема проектировалась courseSlug-aware с Ф0 (все учебные таблицы уже несут `courseSlug` с дефолтом `ai-basics`), поэтому мультикурс — это «перестать хардкодить», а не миграция данных. Все существующие строки — `ai-basics`, коллизий нет; правка уникального индекса безопасна.
**Альтернативы:** глобально-уникальные lessonId через префикс курса (ломает контент-контракт и существующие данные); отдельная таблица Course в БД (дублирует course.yaml, против D-002). Отвергнуто.

### D-037. Юр-блок, куки-баннер, Resend Broadcasts, дизайн-свет, контактный email
**Решение (пакетно):**
- **Куки-баннер информационный, не блокирующий (ревизия UX-07):** платформа ставит только строго необходимый сессионный cookie (D-008) и cookieless-аналитику Plausible → по ePrivacy/GDPR консент-стена не требуется. Баннер = краткий текст + ссылка на `/cookies` + «Понятно», выбор в `localStorage`, никаких трекинг-куки. Минимальный клиентский JS (UX-01, стек). [РЕШЕНИЕ АВТОРА — подтвердить: информационный баннер, а не consent-gate.]
- **Юр-страницы** `/privacy`, `/terms`, `/cookies` — структура-заглушка с TODO, тексты от Ивана, ссылки в футере.
- **Resend Broadcasts:** живой синк подписчиков в Audience (`RESEND_AUDIENCE_ID`); `User.resendContactId` для точечного удаления; сбой синка не роняет основную операцию (зеркало правила D-014, не второй источник правды).
- **Дизайн-свет:** светящиеся CTA (референс wolfai.be), градиентные рамки, slow-gradient hero, MUBI-кадры — CSS-only, в палитре дизайн-брифа, prefers-reduced-motion (расширяет UX-05, дизайн-бриф §6).
- **Навигация на русском** (NAV-01); контактный email → `edu@cratstudio.com` в ru.ts (NAV-02), обновляет D-030 (был временно `edu@skld.me`; зависимость: Иван заводит ящик).
**Причина:** прямые пункты ТЗ Ивана 2026-07-10; юр-блок и явные согласия обязательны для сбора ПД и рассылки; свет-слой — визуальное дозревание брифа; русская навигация — правило UX-02.
**Альтернативы:** полноценный consent-management (Cookiebot и т.п.) — избыточен без трекинг-куки, отвергнут; хранить статус подписки в Resend как источник правды — отвергнуто (D-014 остаётся источником).

## 2026-07-09 — Решения фазы Ф1 (реализация)

### D-030. Дизайн-слой (Ф5) — досрочно; бренд-наследование SKLAD → CRAT studio (ТЗ Ивана 2026-07-09)
**Решение:** визуальная система выполняется сейчас, до Ф3-завершения (Ф3 приостановлена после T1 на своей ветке — её страницы строятся уже в новом стиле). Источник правды — docs/design-brief.md: тексты Ивана финальные (SITE-05 закрыт для главной/лендинга), команда со SKLAD-лендинга БЕЗ Валерии Таран, публичный бренд только CRAT studio (SKLAD — фактура, не бренд), контактный email временно edu@skld.me через словарь. Без Tailwind/Framer Motion — CSS-only.
**Причина:** прямое указание Ивана («это очень важно»); ранний дизайн-слой избавляет Ф3-Ф4-страницы от двойной переделки.
**Альтернативы:** доделать Ф3 сначала — отвергнуто Иваном по приоритету; отдельная /about — отложена (секция на главной).

### D-029. Каноничный счётчик «12/12» для сертификата — живое определение D-004
**Решение:** условие CERT-01 («все 12 уроков пройдены») проверяется по живому определению quizPassedAt && practiceDoneAt на момент проверки — то же, что видит студент в прогрессе. `completedAt` — только момент ПЕРВОГО достижения (тайминг deferred-повторения), на сертификат не влияет и не откатывается.
**Причина:** правило 9 — одно определение «пройден» везде; выданный сертификат при снятом чекбоксе не отзывается (E16), но НОВЫЙ сертификат при 11 живых галочках не выдаётся.
**Альтернативы:** считать по completedAt — дало бы сертификат при фактически снятой практике; отвергнуто.

### D-028. В email_log не хранятся сырые токены входа; переотправка = новый токен (усиление D-009)
**Решение:** payload письма в email_log не содержит magic-link URL. Ручная переотправка (ADM-08) не воспроизводит старое письмо байт-в-байт, а выпускает НОВЫЙ токен и собирает свежую ссылку.
**Причина:** дамп базы не должен давать рабочие ссылки входа (та же логика, что D-009 для таблицы magic_links); бонус — переотправленное письмо всегда содержит живую ссылку, а не истёкшую 15-минутную.
**Альтернативы:** хранить URL в payload ради точного повтора письма — отвергнуто: точный повтор не нужен, живая ссылка полезнее.

### D-027. Сессионный токен без версионирования и ротации секрета (дополнение к D-008)
**Решение:** формат cookie фиксированный (payload.hmac), поддержки двух секретов и версии токена нет. SESSION_SECRET — fail-fast при отсутствии/короче 32 символов в проде.
**Причина:** TTL 30 дней сам «лечит» смену формата или секрета — все токены переиздаются за месяц; смена секрета = массовый разлогин, что при инциденте и требуется. Строить ротацию сейчас — преждевременно.
**Альтернативы:** версия в токене + dual-secret — вернуться, если появится требование отзыва отдельных сессий.

## 2026-07-08 — Расширение видения платформы (утверждено Иваном в сессии брейнсторма)

### D-018. Урок — MDX с белым списком компонентов (контракт контента v2)
**Решение:** `lesson.mdx` вместо `text.md`: markdown + компоненты Figure, Gallery, Callout, Video, Download, Trainer, Animation, Divider (docs/content-format.md §4). Произвольный JS/JSX и компоненты вне списка — ошибка валидации. Анимации — только из библиотеки платформы по id.
**Причина:** Иван требует «крутую современную вёрстку» уроков с иллюстрациями, анимациями и вставками тренажёров; простой markdown этого не даёт. Белый список сохраняет главное свойство контракта: фабрика пишет просто, платформа гарантирует вид и валидацию.
**Альтернативы:** блоки в YAML — тяжело писать длинные тексты; чистый markdown — не покрывает видение. Согласовано с course-factory 2026-07-08 (по поручению Ивана; записи в DECISIONS.md и CLAUDE.md фабрики).

### D-019. Квиз открывается кнопкой «Завершить урок»
**Решение:** квиз — отдельный шаг после контента урока, запускается кнопкой «Завершить урок» (LES-01); определение «пройден» не меняется (квиз ≥2/3 И практика, D-004).
**Причина:** явный ритуал завершения по видению Ивана; отделяет чтение от проверки, в Ф5 ложится на «сцену с прожектором».

### D-020. Каталог тренажёров — только студентам, без геймификации (выбор Ивана)
**Решение:** `/app/trainers` в кабинете, только с enrollment; публичной витрины нет; очки/бейджи/коллекции не делаем. Тренажёры также встраиваются в уроки `<Trainer>` с общими лимитами студента (TRN-06, TRN-07).
**Альтернативы:** публичная витрина как лид-магнит, публичные тренажёры с лимитами, коллекция-полка — отвергнуты Иваном на этом этапе; можно вернуться отдельной фазой.

### D-021. Статьи `/articles` на том же MDX-движке; 5–10 статей к запуску (выбор Ивана)
**Решение:** статья = каталог в `content/articles/` (meta.yaml + article.mdx), публикация = git push, `<Trainer>` в статьях запрещён. К публичному запуску — 5–10 статей; статьи пишет course-factory (решение Ивана 2026-07-08).
**Причина:** SEO/GEO двигается контентом, двух посадочных страниц мало для конкурентных запросов; отдельная CMS дублировала бы git-процесс (в логике D-002).

### D-022. SEO/GEO-пакет: база в Ф1, полный — в Ф6
**Решение:** SSR всех публичных страниц, Metadata API + OG, sitemap/robots — с Ф1; schema.org JSON-LD, `/llms.txt` для нейропоисковиков, CWV-аудит, Search Console + Яндекс.Вебмастер — Ф6 (раздел 14 требований).
**Причина:** база почти бесплатна при постройке страниц, а переделка задним числом дорога; полный пакет имеет смысл при появлении статей.

### D-023. Google Drive (`edu@skld.me`): вход для материалов и выход для документов; раздача файлов — только с нашего сервера
**Решение:** исходники материалов команда ведёт в Drive; в git они попадают скриптом синка. Прямые ссылки на Drive в контенте запрещены. В Ф8 админка складывает копии сертификатов/экспортов в Drive через service account.
**Причина:** Drive — удобное место работы команды, но плохой хостинг для студентов (лимиты, «файл недоступен», нет контроля доступа из ЛМС).
Подтверждено Иваном 2026-07-08: скрипт синка живёт в репозитории course-factory (контент производится там, платформа получает готовое через git push), не в crat-studio.

### D-024. Stripe — Ф7: Checkout + webhook + авто-enrollment (уточнение Ивана к D-001)
**Решение:** после запуска ядра подключаем Stripe Checkout на лендинге: оплата → webhook (проверка подписи, идемпотентность) → User + Enrollment `source=stripe` + письмо со входом; ручная выдача остаётся. Используется существующий аккаунт Ивана (подключение к Тильде не мешает — отдельные ключи и webhook).
**Причина:** D-001 («без Stripe в MVP») остаётся в силе для Ф0–Ф5; поле `Enrollment.source` было заложено именно под это.

### D-025. Инфраструктура: GitHub Actions CI, Sentry, UptimeRobot; сознательный отказ от Redis/S3/CMS/очередей
**Решение:** добавляем только три бесплатных сервиса: CI (typecheck, lint, gitleaks), мониторинг ошибок, мониторинг аптайма `/health`. Redis, S3/R2, CMS, очереди не вводим.
**Причина:** на масштабе одного инстанса и файлового контента это лишние сущности. Триггер пересмотра: если ассеты контента заметно раздуют git-репозиторий — рассмотреть Cloudflare R2.

## 2026-07-08 — Стартовые решения (зафиксированы в брифе Ивана)

### D-001. Без Stripe в MVP
**Решение:** оплат в платформе нет; доступ выдаёт администратор вручную. В `Enrollment` заложено поле `source` (сейчас `manual`).
**Причина:** первый поток — ручные продажи; интеграция платежей не окупается до проверки спроса. Поле `source` позволяет добавить Stripe без миграции смысла данных.
**Альтернативы:** Stripe Checkout сразу — отвергнуто как преждевременная сложность.

### D-002. Контент — файлами из course-factory, не в базе
**Решение:** платформа рендерит файлы из `content/` (yaml + markdown + pdf); обновление урока = git push. Контракт — docs/content-format.md.
**Причина:** контент производится в отдельном репозитории course-factory с собственным процессом ревью; версионирование через git, никакой админки редактирования контента не нужно.
**Альтернативы:** CMS / контент в Postgres — отвергнуто: дублирует git-процесс фабрики, добавляет UI и миграции ради нуля пользы.

### D-003. Magic link вместо паролей
**Решение:** единственный способ входа (и студента, и админа) — одноразовая ссылка на email. TTL 15 минут, строго одноразовая. Сессия 30 дней, httpOnly cookie.
**Причина:** аудитория без технической подготовки — пароли это забытые пароли, ресеты и поддержка. Email и так обязателен.
**Альтернативы:** пароли, OAuth (Google) — отвергнуто: лишние шаги и внешние аккаунты для аудитории 40+.

### D-004. «Урок пройден» = квиз ≥2/3 И практика отмечена
**Решение:** единственное определение пройденности, оно же используется для сертификата. Просмотр видео/текста на статус не влияет.
**Причина:** проверяемые сигналы усвоения; просмотр видео не измерить честно. Одно определение — нет расхождений между прогресс-баром и сертификатом.
**Альтернативы:** учитывать просмотр видео, вводить проценты — отвергнуто как неизмеримое и усложняющее.

### D-005. Отложенные квизы вычисляются при входе, без cron
**Решение:** при загрузке кабинета платформа находит уроки, пройденные ≥7 дней назад, без сданных отложенных вопросов, и показывает блок «3 вопроса на 2 минуты».
**Причина:** cron на Render — отдельная сущность с отдельными сбоями; вычисление при входе даёт тот же продуктовый эффект (студент видит блок, когда приходит) при нулевой инфраструктуре.
**Альтернативы:** cron + push-письма «пора повторить» — можно добавить позже, письма потребуют согласия на рассылку.

### D-006. Админ определяется через ADMIN_EMAILS
**Решение:** админ = пользователь, чей email входит в env-переменную `ADMIN_EMAILS`; учётки синхронизируются при старте приложения (seed), вход тем же magic link.
**Причина:** админов 1–2 человека; UI управления ролями — лишний код и лишняя поверхность атаки. Env-переменная меняется без деплоя кода.
**Альтернативы:** таблица ролей с UI — преждевременно.

## 2026-07-08 — Решения, принятые при написании спецификаций

### D-007. Платформа — отдельный репозиторий `crat-studio`
**Причина:** CLAUDE.md course-factory явно запрещает код продукта в репо фабрики. Подтверждено Иваном. (2026-07-08: папка переименована из `crat-platform` в `crat-studio` по просьбе Ивана.)

### D-008. Сессии — stateless-cookie, без таблицы sessions
**Решение:** сессия — зашифрованная/подписанная httpOnly cookie (SESSION_SECRET), внутри userId + expiry; на каждом запросе пользователь читается из базы.
**Причина:** Иван перечислил точный список таблиц — sessions в нём нет; GDPR-удаление автоматически инвалидирует сессии (пользователя больше нет). Минус — нельзя отозвать одну конкретную сессию досрочно; для MVP приемлемо.
**Альтернативы:** таблица sessions — даст отзыв сессий, добавить при необходимости.

### D-009. Токены magic link хранятся только хэшем (SHA-256)
**Причина:** утечка базы не должна давать рабочие ссылки входа. Сырой токен существует только в письме.

### D-010. Сертификат переживает GDPR-удаление в обезличенном виде
**Решение:** при GDPR-удалении каскадно удаляются все данные студента, кроме строки `Certificate`: в ней обнуляются `userId` и `fullName`, статус → `REVOKED`. `/cert/{номер}` показывает «сертификат отозван по запросу владельца». Номер не переиспользуется.
**Причина:** прямое требование брифа; строка без ПД не нарушает GDPR, а публичная проверка остаётся честной.

### D-011. PDF сертификата генерируется по требованию, не хранится
**Решение:** PDF рендерится Playwright'ом из данных при выдаче (для письма) и при каждом скачивании из кабинета; в базе — только метаданные.
**Причина:** диск Render эфемерен без платного volume; данные для рендера всегда есть, пока сертификат действителен. После GDPR-удаления PDF не восстановим — и не должен быть.

### D-012. Отложенные вопросы = те же 3 вопроса квиза урока
**Решение:** если в quiz.yaml нет необязательного блока `deferred`, спустя 7 дней задаются те же 3 вопроса. Результат пишется в `deferred_quiz_state` и НЕ влияет на статус «пройден».
**Причина:** интервальное повторение работает и на тех же вопросах; не блокируем платформу ожиданием дополнительного контента от фабрики. Блок `deferred` в контракте оставлен на будущее. [РЕШЕНИЕ АВТОРА — подтвердить у Ивана]

### D-013. Ретраи писем — в процессе, без внешней очереди
**Решение:** отправка сразу; при ошибке до 3 повторов с бэкоффом (1 мин / 5 мин / 15 мин) внутри Node-процесса; всё логируется в `email_log`. Потерянные при рестарте ретраи остаются в статусе FAILED и видны в админке для ручной переотправки.
**Причина:** Render web service — долгоживущий процесс, in-process таймеры работают; очередь (BullMQ + Redis) — лишняя инфраструктура для трёх типов писем.

### D-014. Согласия — append-only журнал
**Решение:** таблица `consents` только добавляется (отписка = новая запись `granted=false`); действующее согласие = последняя запись по типу. Ключ — email (согласие даётся до создания User).
**Причина:** GDPR-аудит требует истории: когда дано, когда отозвано, из какого источника.

### D-015. Rate limiting — в памяти процесса
**Решение:** лимиты на регистрацию (5/час с IP) и запрос magic link (3/15 мин на email) — in-memory (LRU). Лимиты тренажёра T1 (20/день, 3/мин на студента) — в базе (`trainer_usage`), т.к. должны переживать рестарт.
**Причина:** один инстанс Render; рестарт сбрасывает счётчики anti-abuse лимитов — приемлемо, это не биллинг. Дневной лимит T1 — это деньги (Anthropic API), поэтому в базе.

### D-016. Мини-проект: каждая пересдача = новая строка `submissions`
**Причина:** сохраняется история попыток и комментариев админа; «текущая» = строка с максимальным `attempt`. Пересдачи не ограничены.

### D-017. Один сайт: главная студии + лендинг курса + ЛМС (уточнение Ивана, 2026-07-08)
**Решение:** это не «только ЛМС», а сайт CRAT studio целиком: `/` — главная студии, `/ai-basics` — лендинг курса с формой заявки, `/app` и `/admin` — учебная часть. Всё — одно Next.js-приложение на одном домене (SITE-01…06).
**Причина:** один деплой, один дизайн, форма заявки живёт прямо на лендинге; отдельный сайт-визитка дал бы два репозитория и рассинхрон стиля.
**Альтернативы:** лендинг на Tilda/отдельном хостинге — отвергнуто: разрыв стиля, форма заявки всё равно должна писать в нашу базу.

### D-018. Ф0-скаффолд: две временные заглушки в package.json (Task 1, 2026-07-08)
**Решение:** (1) `"lint": "eslint"` вместо `"next lint"` — команда `next lint` удалена из CLI в установленной версии Next.js (16.2.10), `create-next-app` сам генерирует `eslint` с flat-конфигом `eslint.config.mjs`. (2) `"build": "next build"` без `prisma generate` — схема Prisma появится в Task 3, до этого `prisma generate` падать не должен и не на чем генерировать клиент.
**Причина:** зафиксировать версию инструментов на момент скаффолда, чтобы не оставлять «сломанный по умолчанию» скрипт незадокументированным.
**Альтернативы:** пин Next.js на 15.x ради соответствия `next lint` — отвергнуто, стек фиксирует мажорный фреймворк, а не конкретный патч; актуальная версия достаточна для MVP. Обновить `build` на `prisma generate && next build` в Task 3.
