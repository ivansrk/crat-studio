## 2026-07-20 — Автозачисление публичной заявки после подтверждения email (D-054)

### D-054. До Stripe публичная регистрация выдаёт доступ сама; админ — наблюдатель

**Решение (владелец Иван, 2026-07-20):** «после того как они записались — им сразу выдавался доступ, но я по-прежнему видел их в админке как новых участников; со Stripe доступ будет выдаваться после оплаты».

**Реализация:** публичный путь `confirmRegistration` (lib/registration/confirm.ts) больше не останавливается на `CONFIRMED` — после клика по ссылке подтверждения выполняется тот же авто-финал, что в инвайт-пути (F14): `User` + `Enrollment(source='site-optin')` + `ENROLLED`, экран с паролем, письмо WELCOME, Resend-синк при подписке. Общий финал вынесен в `finishAutoEnroll`. Участник виден админу: строка «зачислен» в `/admin` (сортировка по свежести — новые сверху) и в `/admin/students`.

- **[РЕШЕНИЕ АВТОРА] Доступ выдаётся после клика подтверждения, а не прямо на сабмите формы.** Причины: (1) согласие на обработку ПД фиксируется только действием самого человека (REG-13) — иначе double opt-in теряет смысл; (2) опечатка в email на сабмите создавала бы мёртвую учётку с паролем, отправленным «в никуда». С TTL 72 часа (D-053) шаг подтверждения перестал терять людей.
- **Ф7/Ф8 (Stripe):** триггером авто-финала станет подтверждённая оплата (webhook), автозачисление по opt-in отключается; `Enrollment.source` уже различает пути (`site-optin` / `manual` / инвайт-лейблы) — миграция не нужна.
- **Ручная выдача (ADM-03) сохранена** для легаси-заявок `CONFIRMED` (подтвердились до этого деплоя).

**Отвергнутые альтернативы:** выдача на сабмите формы (см. выше); фича-флаг в env (лишняя ручка — переключение придёт вместе с кодом Stripe в Ф8).

## 2026-07-20 — Переотправка double opt-in из админки + TTL OPT_IN 72 часа (D-053)

### D-053. Заявки перестают теряться на шаге подтверждения email

**Проблема (боевая):** двум реальным заявителям нельзя было выдать доступ — заявки застряли в `PENDING_OPT_IN`. Причины системные: (1) токен подтверждения жил 60 минут (общий `RESET_TTL_MS` с reset-токеном) — аудитория 40+ открывает почту позже, ссылка мертва; (2) переотправить `DOUBLE_OPT_IN` из админки было нельзя (`unsupported_type` в resend-email), а про «отправьте форму ещё раз» людям никто не сообщает.

**Решение:**
- **TTL по назначению токена:** `OPT_IN` — 72 часа (`OPT_IN_TTL_MS`), `PASSWORD_RESET` — прежние 60 минут (короткий TTL важен: reset-ссылка даёт вход в учётку; opt-in-ссылка лишь подтверждает email, риск ниже, простой TTL терял людей). REG-11 дополнен.
- **ADM-14:** у заявок `PENDING_OPT_IN` в `/admin` появилась кнопка «Отправить подтверждение ещё раз» — новый OPT_IN-токен + новое письмо `DOUBLE_OPT_IN` (новая запись email_log, MAIL-05). Доменная функция `lib/admin/resend-opt-in.ts`.

**Отвергнутые альтернативы:**
- *Ручное подтверждение админом («считать email подтверждённым»)* — фабрикует согласие на обработку ПД: `Consent` пишется только по клику самого человека (REG-13). Юридически слабо — отказ.
- *Переотправка через раздел «Письма» (resendFromLog)* — исходный токен-URL в payload не хранится (D-028), а тип DOUBLE_OPT_IN там сознательно не поддержан; кнопка у самой заявки очевиднее для админа, чем поиск письма в логе.
- *Автоматический ре-отправитель по крону* — инфраструктуры кронов в MVP нет; ручная кнопка решает боль сейчас.

[РЕШЕНИЕ АВТОРА] TTL 72 часа выбран без согласования (баланс «человек успевает» / «токен не вечный», в рамках 7-дневного окна REG-15) — Иван может скорректировать одним числом в `lib/auth/reset.ts`.

## 2026-07-16 — Пакет навигации кабинета студента (D-051)

### D-051. Пакет навигации кабинета студента: меню уроков, возврат из тренажёра, h1 каталога (nav-lessons, 2026-07-16)
**Решение (владелец Иван, аудит навигации 2026-07-16, «вноси все» — находки S1/S5/S7/S9):** только визуально-навигационный слой поверх существующих данных Ф2–Ф4, без новых сущностей/таблиц/бизнес-логики.
- **S1/S9 — меню уроков на странице урока (NAV-03/04).** Новый серверный компонент `components/site/LessonNav.tsx` рендерит на ≥1024px тихий боковой **рейл** (модули — mono-кикеры, уроки списком; текущий выделён + `aria-current="page"`, пройденные — мятная галочка), а на <1024px — компактную липкую **подшапку** урока с дисклоужером «Уроки» (тот же список оверлеем), мини-контекстом «Урок N из M» и компактными prev/next (закрывает S9 — не мотать страницу урока до нижнего пейджера). Данные — программа `course.yaml` + статус «пройден» из `lib/progress` (переиспользован `isLessonPassed`, тот же критерий LES-12, что на курсовой странице; на странице урока добавлен один вызов `getCourseProgress`). CSS-only: рейл сворачивается **чекбокс-хаком** (по умолчанию свернут на 1024, развёрнут на ≥1280 — инверсия дефолта через медиазапрос), мобильное меню — нативный `<details>`; оба no-JS-friendly, движение глушится глобальным `prefers-reduced-motion`. Прыжок между дальними уроками (напр. 1.1→3.2) — 2 клика (раскрыл список → выбрал урок) вместо возврата на курсовую страницу.
- **S5 — возврат из тренажёра в урок (NAV-05).** `<Trainer>` со страницы урока добавляет `?from={lessonId}` (lessonId прокинут в `mdxComponents(base, lessonId)`); страницы `t1/t2/t3` через общий `TrainerFooterNav` показывают «Вернуться к уроку "{название}"» рядом с «К тренажёрам». `from` валидируется по контенту (`getLesson('ai-basics', from)`), href строится из самого урока — не открытый редирект; невалидный/отсутствующий `from` → только «К тренажёрам».
- **S7 — видимый h1 каталога тренажёров (NAV-06).** `/app/trainers` открывается `<h1 class="crat-display">` из `t.trainers.catalogTitle` + короткий подзаголовок — раньше был только mono-кикер без заголовка первого уровня.
**Причина:** взрослой аудитории (40+) нужен доступ ко всей программе прямо из урока (иначе — возврат на курсовую страницу за каждым переходом) и явный выход из тренажёра обратно в контекст; отсутствие h1 на каталоге — дыра в структуре страницы (UX-02/доступность). Всё вписано в тихий интерфейс брифа: рейл в muted-тонах, красный — только текущий урок, мята — отметка «пройден».
**Альтернативы:** превратить страницу урока в IDE-подобный трёхколоночник — отвергнуто (бриф «одна страница = одно действие», проза сохраняет ≤75ch); клиентский JS-аккордеон — отвергнуто (CSS-only, no-JS-friendly); дублировать список уроков разметкой курсовой страницы — отвергнуто (общий презентационный компонент, данные из одного источника). Смещения sticky-подшапки привязаны к высоте `CabinetHeader` (переносится на мобильном) медиазапросами — замер 2026-07-16; при смене шапки offset пересмотреть. Нумерация: параллельная ветка nav-admin-articles ведёт независимую запись D-051b (админка/статьи) — разные разделы, конфликта содержания нет.

### D-051c. Навигация кабинета: кликабельное название курса, редирект единственного курса, h1 и «Продолжить» на странице курса (nav-cabinet, 2026-07-16)
**Решение (владелец Иван, аудит навигации 2026-07-16, находки S2/S3/S4/S8 — финальная часть пакета D-051):** визуально-навигационный слой поверх данных Ф2–Ф4, без новых сущностей/бизнес-логики.
- **S2 — название курса в шапке кликабельно (NAV-07).** `CabinetCourseTitle` рендерит `Link` на `/app/{slug}` (не `<span>`), когда курс известен по pathname; на страницах кабинета без курса — по-прежнему null (ссылки нет). Стиль тихий: цвет `--paper-muted`, подчёркивание фоновым градиентом (reveal-line-паттерн, не `text-decoration`) только под `@media(hover:hover)`, `:focus-visible` — neon-outline. **Высота `CabinetHeader` не изменилась** (замер до/после на 360 и 1440 — см. отчёт задачи): inline-ссылка того же размера, что был span; sticky-offset рейла/подшапки урока (D-051, NAV-03/04) не тронут.
- **S3 — «домой» для студента с одним курсом (NAV-08).** `soleCourseRedirect(mine)` теперь редиректит `/app`→`/app/{slug}` при **ровно одном** enrollment на опубликованный курс, **не** требуя пустого каталога остальных (раньше demo-course «Скоро» держал студента на лишнем экране-хабе). Логотип шапки ведёт на `/app` → редирект в курс = «домой». Хаб/каталог не потеряны: тихая ссылка **«Все курсы»** над h1 страницы курса ведёт на `/app?all=1`, и хаб пропускает редирект при наличии `all` (иначе ссылка зациклилась бы обратно в курс). Единственный **неопубликованный** курс редирект не вызывает (иначе `/app/{slug}` → 404).
- **S4 — различить хаб и курс (NAV-09).** h1 страницы курса `/app/{slug}` = название курса (было общее «Кабинет» на обеих страницах). Хаб `/app` сохраняет h1 «Кабинет» (у него уже есть h2 «Мои курсы» — дубля нет). Крупный display-h1 и мелкий mono-кикер названия в липкой шапке не конфликтуют.
- **S8 — «Продолжить: урок N» (NAV-09).** Заметная primary-кнопка на первый непройденный урок, рядом с горизонтом прогресса, до списка модулей. Домен-логика «первый непройденный» — чистая функция `firstUnpassedLesson(lessons, byLesson)` в `lib/progress/quiz-logic.ts` (рядом с `isLessonPassed`, тот же критерий LES-12; с тестами). **Поведение у дипломанта (все уроки пройдены): кнопки нет** — `firstUnpassedLesson` возвращает null; следующий шаг это финальный проект, чей блок с CTA уже виден ниже (лидировать «Продолжить проект» рядом с уже-заметным блоком проекта — дубль).
**Причина:** взрослой аудитории (40+) нужен один явный путь обратно в курс и «дом», совпадающий с ожиданием (лого → мой курс, а не пустой хаб). Тихий интерфейс брифа сохранён: название-ссылка muted, «Все курсы» — mono-крошка, красный — только primary-кнопка «Продолжить».
**Альтернативы:** редирект-хаба при пустом каталоге (прежнее условие) — отвергнуто (demo-course «Скоро» его блокирует, студент всегда видит лишний экран); «Продолжить» ведёт на финальный проект при полном прохождении — отвергнуто (дублирует уже-заметный блок проекта); держать логику «первый непройденный» инлайн в page.tsx — отвергнуто (домен-правило LES-12 должно тестироваться чистой функцией).

## 2026-07-15 — Исходящий FROM писем на собственный ящик (D-046-амд)

### D-046-амд. Исходящий FROM писем → edu@cratstudio.com
**Решение (амендмент к D-046):** `FROM` в `lib/email/index.ts` переключён с `hello@cratstudio.com` на `edu@cratstudio.com`. Прежний D-046 сознательно не трогал FROM, оставляя `hello@`, — но ящика `hello@` на домене нет, поэтому письма double opt-in/WELCOME/сертификат уходили «с несуществующего адреса» (жалоба Ивана «дойдут ли письма double opt-in?»). Ящик `edu@cratstudio.com` заведён, домен `cratstudio.com` в Resend верифицирован — письма с него ходят.
**Живая проба на этой машине не выполнена:** в локальном `.env` нет `RESEND_API_KEY` (только `DATABASE_URL` и `ADMIN_EMAILS`), одноразовый пробник (renderEmail DOUBLE_OPT_IN → Resend, FROM=edu@) вернул `Missing API key`. Отправку с боевым ключом нужно проверить на Render/с ключом (см. отчёт задачи). Цепочка double opt-in по коду проверена: `/register`→PENDING_OPT_IN→письмо DOUBLE_OPT_IN со ссылкой `${APP_URL}/invite-confirm/{token}`→CONFIRMED+Consent; ссылка строится от `APP_URL` (прод — cratstudio.com).

## 2026-07-15 — Удаление участника отовсюду единой функцией (D-050)

### D-050. Единая функция удаления участника; сертификат удаляется физически; email_log обезличивается
**Решение (по прямой жалобе Ивана «снова не могу удалять участников» + «удалять отовсюду, чтобы удалялось везде»):**
- **Одна доменная функция** `deleteParticipant(ref, confirmEmail)` (`lib/admin/delete-participant.ts`, заменяет `gdprDeleteStudent`/`lib/admin/gdpr.ts`). `ref` — `{ userId }` (студент/клиент) ИЛИ `{ registrationId }` (заявка-лид). Функция разрешает email + опциональный userId (заявка могла уже стать учёткой с тем же email — удаляется и она), сверяет `confirmEmail` (ADM-11) и `isAdminEmail` (нельзя удалить админа), затем в одной транзакции покрывает ВСЕ связанные таблицы по schema.prisma.
- **«Удалять отовсюду» (ADM-13):** кнопка «Удалить» добавлена в список заявок `/admin`, список и карточку студентов, список и карточку клиентов. Все вызывают `deleteParticipantAction` → одну функцию. Удаление из любой точки убирает человека везде. UI без клиентского JS — нативный `<details>` + ввод email участника (общий компонент `components/admin/DeleteParticipant.tsx`).
- **Сертификат — ФИЗИЧЕСКИ удаляется** (отменяет REVOKED-обезличивание D-010/CERT-07): право субъекта на удаление ПД приоритетнее реестра выданных документов. После удаления `/cert/{номер}` честно отвечает «не найден» (та же ветка, что несуществующий номер — app/cert/[number] уже так устроен). Счётчик `CertificateCounter` не откатывается → номера по-прежнему не переиспользуются (CERT-03).
- **EmailLog — обезличивается** (userId→null, toEmail→`[удалён]`), а не удаляется каскадом: журнал доставки без связи с человеком и без адреса — не персональные данные, но полезен как факт отправки. Реализация: `updateMany` ставит userId=null ДО `user.delete`, тем самым выводя строки из-под `onDelete: Cascade` (по userId и по адресу — письма double opt-in уходят до создания User).
- **ConsultationRequest — обезличивается** (userId→null), контакт-лид остаётся у студии (без изменений, D-048/D-036).
- **Подтверждение — единый typed-email** во всех точках (ADM-11): аудитория-админ одна, но удаление необратимо и в списках легко промахнуться строкой — ввод email конкретного участника защищает от удаления не того. Это тот же паттерн, что уже был в «Опасной зоне» карточки студента; чекбокс отвергнут как более слабая защита при удалении из списка.
**Причина:** до этого удаление жило ТОЛЬКО в «Опасной зоне» карточки студента (T8), а заявки-лиды без учётки нельзя было удалить нигде — отсюда жалоба «не могу удалять». Разрозненные точки удаления и «сертификат переживает удаление» противоречили прямому требованию Ивана «удалять отовсюду и чтобы удалялось везде».
**Отвергнуто:** оставить REVOKED-сертификат ради публичного реестра — отклонено Иваном (GDPR приоритетнее); физически удалять и email_log — отклонено (лог доставки без ПД полезен для отладки/аудита отправки, обезличивания достаточно); отдельные функции удаления под каждый раздел — отклонено (рассинхрон и дубли логики, прямое нарушение «одна функция»).

### D-039. unpublished блокирует доступ ЖИВЫМ enrollment — риск для оплативших [РЕШЕНИЕ АВТОРА]
**Найдено (ревью Ф7в, финал-ревью M-блока):** `published` (D-036, `course.yaml`) на false закрывает не только каталог/лендинг, но и все маршруты кабинета для курса — `/app/{courseSlug}`, `/app/{courseSlug}/lessons/{lessonId}`, `/quiz`, `/project`, `/certificate` — все делают `if (!entry.published) notFound()` (app/app/[courseSlug]/**, app/actions/{lesson,quiz,project}.ts) без проверки, есть ли у пользователя живой `Enrollment`. Сейчас это безопасно только потому, что курс всего один и снятие с публикации не практиковалось.
**Дефект по существу:** сегодняшняя реализация — «unpublished = курс не существует» — недопустима для студента, который уже оплатил/получил доступ (Enrollment существует): администратор, сняв курс с публикации (например, для правки контента), тем самым отбирает доступ у уже обучающихся людей без предупреждения — 404 вместо урока.
**Решение сейчас (зафиксировать явно, не откладывать молча):** оставляем текущее поведение (unpublished = блокирует всех, включая enrolled) как временное — при одном курсе в проде цена ошибки низкая, а другого использования `published` в MVP не было. **Правило на будущее:** при появлении второго реального курса (или до первого фактического снятия курса с публикации на проде, что раньше) — пересмотреть: `published=false` должен(а) скрывать курс из каталога/`/app` и запрещать НОВЫЕ Enrollment, но не разрывать доступ уже зачисленным студентам к урокам/квизам/сертификату, которые они оплатили или получили.
**Причина отложить, а не чинить сейчас:** курс один (`ai-basics`), живых кейсов «сняли с публикации при активных студентах» не было и не запланировано в ближайших фазах; проверка `Enrollment` в 5 местах — не бесплатный рефакторинг, добавлять его без реального сценария использования — YAGNI. Но риск реален и не должен молчать в коде без записи.
**Альтернативы:** чинить сразу (проверка Enrollment в каждом published-гейте) — отклонено сейчас как преждевременное для однокурсового MVP, но не отклонено навсегда — см. правило выше.

### D-031. Вход по паролю вместо magic link (разворот D-003)
**Решение:** единственный способ входа студента и админа — email + пароль. Magic-link-вход отменяется (AUTH-01/02, flow F3 снимаются). Token-механика magic link (одноразовый SHA-256-хэш, TTL, атомарное гашение `usedAt`) переиспользуется как ссылка восстановления/установки пароля (TTL увеличен до 60 мин) и подтверждения регистрации (double opt-in). Модель `MagicLink` переименовывается в `PasswordResetToken` (+`purpose`).
**Причина:** прямое решение Ивана 2026-07-10. Пароль даёт мгновенный повторный вход без хождения в почту на каждый заход, привычен, работает при проблемах с доставкой писем и нужен как основа для инвайт-авто-выдачи и Stripe.
**Что теряем (честно):** (1) исчезает главное удобство magic link для аудитории 40+ — «никаких паролей»; появляются забытые пароли и обращения в поддержку (частично лечится reset-ссылкой и генерируемым паролем). (2) Пароль — новый секрет, который надо безопасно хранить (хэш) и не светить в логах/письмах (открытый пароль только в письме WELCOME один раз). (3) Нужен миграционный путь для существующих юзеров (D-034). (4) Растёт поверхность атаки (перебор пароля → AUTH-20). **Митигируется:** генерируемый сервером пароль (не слабый пользовательский), reset как «первый вход», rate-limit перебора.
**Альтернативы:** оставить magic link (отвергнуто Иваном); пароль + опциональный magic link параллельно (отвергнуто — две auth-поверхности, сложнее); OAuth-Google (отвергнуто ранее D-003, аудитория без Google-аккаунтов). Обновить CLAUDE.md (принцип «вход без паролей» → «вход по паролю»).

### D-032. Хэш пароля — bcryptjs (не argon2)
**Решение:** пароли хэшируются `bcryptjs` (чистый JS, cost-фактор ≥ 12).
**Причина:** Render собирает образ на каждый деплой; `argon2` (npm) требует нативной сборки (node-gyp/prebuilt-бинарь) — риск падения билда, особенно рядом с уже хрупким Playwright/Chromium-шагом (Ф3). `bcryptjs` — pure-JS, ноль нативных зависимостей, детерминированный деплой; CLAUDE.md уже помечает его «допустимой лёгкой» зависимостью. Наш пароль генерируется сервером (12+ симв., высокая энтропия), поэтому криптографическое превосходство argon2 против слабых пользовательских паролей здесь не решающее; bcrypt cost 12 достаточен.
**Рассмотрено:** node:crypto `scrypt` (встроен, ноль зависимостей вообще) — валидная альтернатива, но ручное управление солью/параметрами и форматом хранения; bcryptjs проще и привычнее команде. `argon2` — вернуться, если появится вход по пользовательскому паролю с высокими требованиями. [РЕШЕНИЕ АВТОРА: bcryptjs; подтвердить у Ивана либо переключить на встроенный scrypt, если хочется совсем без внешних зависимостей.]

### D-033. Сервер генерирует человекочитаемый пароль
**Решение:** при создании учётки сервер генерирует пароль ≥ 12 символов, читаемый (группы вида `xxxx-xxxx-xxxx` из набора без неоднозначных символов 0/O/1/l/I), показывает один раз на экране успеха и шлёт письмом WELCOME. Пользователь может сменить его в кабинете.
**Причина:** пользователь не придумывает пароль на регистрации (одна страница = одно действие, UX-01); генерируемый пароль сильнее и его можно сразу отправить письмом как «первый вход». Читаемый формат — чтобы можно было ввести вручную с телефона.
**Альтернативы:** пользователь задаёт пароль на форме (лишний шаг и слабые пароли); ставить пароль только через reset после первого письма (лишний цикл). Отвергнуто.

### D-034. Миграция существующих пользователей на пароли
**Решение:** `passwordHash` — nullable. Существующие юзеры (созданные до Ф7) имеют `passwordHash=null`. Путь входа: (1) постоянная ссылка «Забыли пароль / первый вход» на `/login` (AUTH-19); (2) разовая рассылка письма PASSWORD_RESET всем таким юзерам после деплоя Ф7а (идемпотентно по `passwordHash=null`). До установки пароля обычный вход невозможен, ответ — то же «неверный email или пароль» (не раскрываем состояние миграции).
**Причина:** на проде уже есть данные; нельзя разлогинить людей без пути назад. Аддитивная nullable-колонка + reset-механика (уже есть) закрывают миграцию без ручной работы.
**Альтернативы:** заставить всех пройти reset без постоянной ссылки (хрупко, если письмо потеряно); генерировать всем новые пароли и разослать (можно, но reset-ссылка безопаснее — пароль не летит в письме массово). Отвергнуто.

### D-035. Инвайт — авто-выдача; публичная заявка — ручная (двойной funnel)
**Решение:** после double opt-in регистрация ПО ИНВАЙТУ выдаёт доступ автоматически (User+пароль+Enrollment, `source=sourceLabel`); ПУБЛИЧНАЯ заявка с `/ai-basics` (без инвайта) после подтверждения email остаётся заявкой — доступ выдаёт админ вручную (ADM-03). Технически оба пути общие (double opt-in, consents, Resend-синк, письма); различие — авто-enroll vs ручной и метка источника.
**Причина:** курс платный, продажи ручные в MVP (D-001) — авто-выдача из открытой формы раздала бы курс бесплатно. Инвайт админ выдаёт осознанно уже оплатившему/приглашённому клиенту → доверие установлено, авто-выдача уместна и убирает ручное бутылочное горло для тёплого трафика.
**Альтернативы:** (а) конвертировать публичную форму в авто-выдачу тоже — отвергнуто (раздача платного курса); (б) убрать публичную форму, оставить только инвайты — отвергнуто (публичный лендинг нужен для лидов/SEO). [РЕШЕНИЕ АВТОРА — подтвердить у Ивана: публичная заявка = ручная выдача, инвайт = авто. До Stripe (Ф8) это основной способ отсекать неоплативших.]

### D-036. Мультикурс — минимально-инвазивно (схема уже courseSlug-aware)
**Решение:** курс = каталог `content/{courseSlug}/`; несколько курсов индексируются `lib/content`. Доменные функции (`lib/progress`, `lib/project`, `lib/cert`, `lib/content`) принимают `courseSlug` параметром вместо константы `COURSE='ai-basics'`. Маршруты становятся `/app/{courseSlug}/…` со 301 со старых. Единственная структурная правка схемы — `QuizResult @@unique` дополняется `courseSlug` (латентный баг: сейчас `(userId, lessonId, attempt)` без курса). `course.yaml` получает необязательный флаг `published`. Формат урока и контракт course-factory (docs/content-format.md) НЕ меняются.
**Причина:** схема проектировалась courseSlug-aware с Ф0 (все учебные таблицы уже несут `courseSlug` с дефолтом `ai-basics`), поэтому мультикурс — это «перестать хардкодить», а не миграция данных. Все существующие строки — `ai-basics`, коллизий нет; правка уникального индекса безопасна.
**Альтернативы:** глобально-уникальные lessonId через префикс курса (ломает контент-контракт и существующие данные); отдельная таблица Course в БД (дублирует course.yaml, против D-002). Отвергнуто.

### D-029. Каноничный счётчик «12/12» для сертификата — живое определение D-004
**Решение:** условие CERT-01 («все 12 уроков пройдены») проверяется по живому определению quizPassedAt && practiceDoneAt на момент проверки — то же, что видит студент в прогрессе. `completedAt` — только момент ПЕРВОГО достижения (тайминг deferred-повторения), на сертификат не влияет и не откатывается.

### D-028. В email_log не хранятся сырые токены входа; переотправка = новый токен (усиление D-009)
**Решение:** payload письма в email_log не содержит magic-link URL. Ручная переотправка (ADM-08) не воспроизводит старое письмо байт-в-байт, а выпускает НОВЫЙ токен и собирает свежую ссылку.
**Причина:** дамп базы не должен давать рабочие ссылки входа (та же логика, что D-009 для таблицы magic_links); бонус — переотправленное письмо всегда содержит живую ссылку, а не истёкшую 15-минутную.
**Альтернативы:** хранить URL в payload ради точного повтора письма — отвергнуто: точный повтор не нужен, живая ссылка полезнее.

### D-027. Сессионный токен без версионирования и ротации секрета (дополнение к D-008)
**Решение:** формат cookie фиксированный (payload.hmac), поддержки двух секретов и версии токена нет. SESSION_SECRET — fail-fast при отсутствии/короче 32 символов в проде.
**Причина:** TTL 30 дней сам «лечит» смену формата или секрета — все токены переиздаются за месяц; смена секрета = массовый разлогин, что при инциденте и требуется. Строить ротацию сейчас — преждевременно.
**Альтернативы:** версия в токене + dual-secret — вернуться, если появится требование отзыва отдельных сессий.

## 2026-07-08 — Стартовые решения (зафиксированы в брифе Ивана)

### D-001. Без Stripe в MVP
**Решение:** оплат в платформе нет; доступ выдаёт администратор вручную. В `Enrollment` заложено поле `source` (сейчас `manual`).
**Причина:** первый поток — ручные продажи; интеграция платежей не окупается до проверки спроса. Поле `source` позволяет добавить Stripe без миграции смысла данных.
**Альтернативы:** Stripe Checkout сразу — отвергнуто как преждевременная сложность.

### D-002. Контент — файлами из course-factory, не в базе
**Решение:** платформа рендерит файлы из `content/` (yaml + markdown + pdf); обновление урока = git push. Контракт — docs/content-format.md.
**Причина:** контент производится в отдельном репозитории course-factory с собственным процессом ревью; версионирование через git, никакой админки редактирования контента не нужно.
**Альтернативы:** CMS / контент в Postgres — отвергнуто: дублирует git-процесс фабрики, добавляет UI и миграции ради нуля пользы.

### D-003. Magic link вместо паролей
**Решение:** единственный способ входа (и студента, и админа) — одноразовая ссылка на email. TTL 15 минут, строго одноразовая. Сессия 30 дней, httpOnly cookie.
**Причина:** аудитория без технической подготовки — пароли это забытые пароли, ресеты и поддержка. Email и так обязателен.
**Альтернативы:** пароли, OAuth (Google) — отвергнуто: лишние шаги и внешние аккаунты для аудитории 40+.

### D-004. «Урок пройден» = квиз ≥2/3 И практика отмечена
**Решение:** единственное определение пройденности, оно же используется для сертификата. Просмотр видео/текста на статус не влияет.
**Причина:** проверяемые сигналы усвоения; просмотр видео не измерить честно. Одно определение — нет расхождений между прогресс-баром и сертификатом.
**Альтернативы:** учитывать просмотр видео, вводить проценты — отвергнуто как неизмеримое и усложняющее.

### D-005. Отложенные квизы вычисляются при входе, без cron
**Решение:** при загрузке кабинета платформа находит уроки, пройденные ≥7 дней назад, без сданных отложенных вопросов, и показывает блок «3 вопроса на 2 минуты».
**Причина:** cron на Render — отдельная сущность с отдельными сбоями; вычисление при входе даёт тот же продуктовый эффект (студент видит блок, когда приходит) при нулевой инфраструктуре.
**Альтернативы:** cron + push-письма «пора повторить» — можно добавить позже, письма потребуют согласия на рассылку.

### D-006. Админ определяется через ADMIN_EMAILS
**Решение:** админ = пользователь, чей email входит в env-переменную `ADMIN_EMAILS`; учётки синхронизируются при старте приложения (seed), вход тем же magic link.
**Причина:** админов 1–2 человека; UI управления ролями — лишний код и лишняя поверхность атаки. Env-переменная меняется без деплоя кода.
**Альтернативы:** таблица ролей с UI — преждевременно.

## 2026-07-08 — Решения, принятые при написании спецификаций

### D-007. Платформа — отдельный репозиторий `crat-studio`
**Причина:** CLAUDE.md course-factory явно запрещает код продукта в репо фабрики. Подтверждено Иваном. (2026-07-08: папка переименована из `crat-platform` в `crat-studio` по просьбе Ивана.)

### D-008. Сессии — stateless-cookie, без таблицы sessions
**Решение:** сессия — зашифрованная/подписанная httpOnly cookie (SESSION_SECRET), внутри userId + expiry; на каждом запросе пользователь читается из базы.
**Причина:** Иван перечислил точный список таблиц — sessions в нём нет; GDPR-удаление автоматически инвалидирует сессии (пользователя больше нет). Минус — нельзя отозвать одну конкретную сессию досрочно; для MVP приемлемо.
**Альтернативы:** таблица sessions — даст отзыв сессий, добавить при необходимости.

### D-009. Токены magic link хранятся только хэшем (SHA-256)
**Причина:** утечка базы не должна давать рабочие ссылки входа. Сырой токен существует только в письме.

### D-010. Сертификат переживает GDPR-удаление в обезличенном виде
**Решение:** при GDPR-удалении каскадно удаляются все данные студента, кроме строки `Certificate`: в ней обнуляются `userId` и `fullName`, статус → `REVOKED`. `/cert/{номер}` показывает «сертификат отозван по запросу владельца». Номер не переиспользуется.
**Причина:** прямое требование брифа; строка без ПД не нарушает GDPR, а публичная проверка остаётся честной.

### D-011. PDF сертификата генерируется по требованию, не хранится
**Решение:** PDF рендерится Playwright'ом из данных при выдаче (для письма) и при каждом скачивании из кабинета; в базе — только метаданные.
**Причина:** диск Render эфемерен без платного volume; данные для рендера всегда есть, пока сертификат действителен. После GDPR-удаления PDF не восстановим — и не должен быть.

### D-012. Отложенные вопросы = те же 3 вопроса квиза урока
**Решение:** если в quiz.yaml нет необязательного блока `deferred`, спустя 7 дней задаются те же 3 вопроса. Результат пишется в `deferred_quiz_state` и НЕ влияет на статус «пройден».
**Причина:** интервальное повторение работает и на тех же вопросах; не блокируем платформу ожиданием дополнительного контента от фабрики. Блок `deferred` в контракте оставлен на будущее. [РЕШЕНИЕ АВТОРА — подтвердить у Ивана]

### D-013. Ретраи писем — в процессе, без внешней очереди
**Решение:** отправка сразу; при ошибке до 3 повторов с бэкоффом (1 мин / 5 мин / 15 мин) внутри Node-процесса; всё логируется в `email_log`. Потерянные при рестарте ретраи остаются в статусе FAILED и видны в админке для ручной переотправки.
**Причина:** Render web service — долгоживущий процесс, in-process таймеры работают; очередь (BullMQ + Redis) — лишняя инфраструктура для трёх типов писем.

### D-014. Согласия — append-only журнал
**Решение:** таблица `consents` только добавляется (отписка = новая запись `granted=false`); действующее согласие = последняя запись по типу. Ключ — email (согласие даётся до создания User).
**Причина:** GDPR-аудит требует истории: когда дано, когда отозвано, из какого источника.

### D-015. Rate limiting — в памяти процесса
**Решение:** лимиты на регистрацию (5/час с IP) и запрос magic link (3/15 мин на email) — in-memory (LRU). Лимиты тренажёра T1 (20/день, 3/мин на студента) — в базе (`trainer_usage`), т.к. должны переживать рестарт.
**Причина:** один инстанс Render; рестарт сбрасывает счётчики anti-abuse лимитов — приемлемо, это не биллинг. Дневной лимит T1 — это деньги (Anthropic API), поэтому в базе.

### D-016. Мини-проект: каждая пересдача = новая строка `submissions`
**Причина:** сохраняется история попыток и комментариев админа; «текущая» = строка с максимальным `attempt`. Пересдачи не ограничены.

### D-017. Один сайт: главная студии + лендинг курса + ЛМС (уточнение Ивана, 2026-07-08)
**Решение:** это не «только ЛМС», а сайт CRAT studio целиком: `/` — главная студии, `/ai-basics` — лендинг курса с формой заявки, `/app` и `/admin` — учебная часть. Всё — одно Next.js-приложение на одном домене (SITE-01…06).
**Причина:** один деплой, один дизайн, форма заявки живёт прямо на лендинге; отдельный сайт-визитка дал бы два репозитория и рассинхрон стиля.
**Альтернативы:** лендинг на Tilda/отдельном хостинге — отвергнуто: разрыв стиля, форма заявки всё равно должна писать в нашу базу.

### D-018. Ф0-скаффолд: две временные заглушки в package.json (Task 1, 2026-07-08)
**Решение:** (1) `"lint": "eslint"` вместо `"next lint"` — команда `next lint` удалена из CLI в установленной версии Next.js (16.2.10), `create-next-app` сам генерирует `eslint` с flat-конфигом `eslint.config.mjs`. (2) `"build": "next build"` без `prisma generate` — схема Prisma появится в Task 3, до этого `prisma generate` падать не должен и не на чем генерировать клиент.
**Причина:** зафиксировать версию инструментов на момент скаффолда, чтобы не оставлять «сломанный по умолчанию» скрипт незадокументированным.
**Альтернативы:** пин Next.js на 15.x ради соответствия `next lint` — отвергнуто, стек фиксирует мажорный фреймворк, а не конкретный патч; актуальная версия достаточна для MVP. Обновить `build` на `prisma generate && next build` в Task 3.

### D-049. Поставка реального контента ai-basics: структура 4+3+3+2, урок 4.3 упразднён, финальный проект = методичка фабрики (2026-07-15)

Финальный проект на платформе — существующая сущность `Submission`; тексты страницы `/app/{slug}/project` (описание задания, форматы, подсказки полей, открытые критерии) и свёрнутая методичка критериев в `/admin/projects` взяты из `final-project/{brief,report-form,criteria}.md` и заведены строками в `lib/i18n/ru.ts` — новых сущностей не вводилось.

Сид обновлён: дипломант проходит все 12 уроков новой структуры, условие сертификата CERT-01 («все 12») выполняется. Контент уроков не редактировался (утверждён фабрикой); валидатор платформы принял пакеты без единой правки формата.
**Альтернативы:** сохранить 4.3 пустой заглушкой ради «ровных» 3×4 — отвергнуто: состав курса определяет фабрика, лишний урок ломал бы условие сертификата и вводил студента в заблуждение. Завести отдельные контент-файлы под тексты проекта — отвергнуто: тексты короткие, страница проекта i18n-driven, длинных полотен нет.

### D-051b. Навигация-связки админки и перелинковка статей (nav-admin-articles, 2026-07-16)
**Решение:** без новых сущностей и таблиц — только навигация поверх существующих данных. (1) Карточка студента `/admin/students/[userId]` получила «← К студентам» и «Карточка клиента» → `/admin/clients/[userId]`; обратная ссылка (карточка клиента → прогресс студента) уже была. (2) В списках заявок (`/admin`) и консультаций (`/admin/consultations`) у человека с учёткой — тихий mono-переход «клиент →» на карточку клиента. Матч заявки — по email (`email` заявки === ключ учётки), один batch-запрос `User.findMany({ email: { in } })`. Матч консультации — по `ConsultationRequest.userId` (CONS-02, надёжная связь из кабинета) с проверкой, что учётка ещё жива (userId без каскада — переживает GDPR-удаление); публичные заявки без userId ссылку не получают, т.к. `contact` — свободный текст (email/телефон/мессенджер), ненадёжен для сопоставления. (3) В admin-nav добавлен «На сайт» → `/` рядом с «Выйти» (первый `margin-inline-start:auto` прижимает пару вправо). (4) Конец статьи `/articles/[slug]` — блок «Читать ещё»: до 3 соседних статей + «← Все статьи». Выбор соседей детерминирован: `getArticles()` уже отсортирован (дата desc, при равенстве slug по алфавиту), берутся идущие следом с закольцовыванием (`[...slice(idx+1), ...slice(0, idx)].slice(0,3)`) — без рандома, стабильно для SSG.
**Причина:** админ ходит между разделами по одному человеку (заявка → клиент → прогресс) — тихие связки убирают ручной поиск по имени; SEO/GEO-разделу нужна внутренняя перелинковка, но «рекомендательная лента» противоречит тихому интерфейсу брифа, поэтому mono-кикер + список ссылок.
**Альтернативы:** матч консультации по email из `contact` — отвергнуто: свободный текст, риск ложного перехода на чужого клиента. Рандомные/по релевантности «похожие статьи» — отвергнуто: недетерминированность ломает SSG-стабильность, полноценная релевантность (теги/эмбеддинги) избыточна для 5–10 статей.
**Примечание о нумерации:** D-051 и D-051b — независимые записи (разные ветки одного пакета); D-051 (при наличии) относится к кабинету студента, D-051b — к админке и статьям. Конфликта содержания нет.

### D-052. Редакционный лонгрид /articles + компоненты статьи v2.1 (editorial-articles, 2026-07-17)
**Решение:** вёрстка статей `/articles/{slug}` переведена в редакционный лонгрид на тёмной системе CRAT (референсы брифа: MUBI/NOWNESS/e-flux, «свежо, современно»), фундамент под будущий генератор контента. Аддитивно к контракту v2 (→ v2.1, `docs/content-format.md` §8):
- **4 новых MDX-компонента только для статей** (`ARTICLE_COMPONENTS` в `lib/content/whitelist.ts`): `<Lead>` (крупный вводный абзац), `<PullQuote cite?>` (вынесенная цитата, красная кавычка-свет, выход за колонку на десктопе), `<KeyPoints>` («Главное за 30 секунд», mono-кикер), `<Sources>` (тихий mono-список источников в конце). Валидатор получил opt-in `extraComponents` (симметрично `forbidComponents`): статьи передают `ARTICLE_COMPONENTS`, уроки — нет, поэтому в `lesson.mdx` эти теги вне белого списка (ошибка). `Figure`/`Gallery` из v2 в статьях уже были разрешены (глобальный белый список) — подтверждено, оставлено.
- **Мета статьи:** `cover` теперь берётся из `public/images/` ИЛИ из `assets/…` статьи (D-052 расширяет D-038); `readingTime` считается платформой из текста (слов/200, ≥1 мин), в мету не выносится (решение владельца); автор не выводится (решение владельца 2026-07-17 «подписывай никак»).
- **Шаблон страницы:** обложка-«кадр» full-bleed со скримом и заголовком поверх (бриф §3 «image is the protagonist»); без обложки — типографическая шапка с крупным заголовком. Дата · время чтения — тихий mono. Тело статьи — скоуп `.article-mdx` (проза с приёмами `.lesson-mdx`: h2 с красной линией-светом, strong/em, списки, blockquote), уроки (`.lesson-mdx`) не затронуты. Оглавление «В этой статье» собирается автоматически из h2 при ≥4 заголовках (`lib/content/toc.ts`: `extractH2`/`slugifyHeading`, id-якоря ставит рендер h2 на странице). `timeRequired` добавлен в JSON-LD статьи (SEO/GEO).
- **Демо-статья** `kak-raspoznat-dipfeyk` получила аккуратную редакторскую правку без изменения смысла: первый абзац → `<Lead>`, выводы → `<KeyPoints>`, яркая фраза-пивот → `<PullQuote>`, `<Sources>` с двумя реальными ссылками (FTC, Europol) — витрина формата для владельца.
**Причина:** статьи — SEO/GEO-канал и фундамент генератора контента; прежняя вёрстка была «голым» MDX без типографики уровня урока. Приёмы урока уже отревьюены и сильны — портированы в свой скоуп, чтобы не трогать кабинет.
**Альтернативы:** (1) единый скоуп для урока и статьи — отвергнуто: у урока «станции» и своя семантика, риск регрессий в отревьюенном кабинете. (2) `rehype-slug`/`rehype-autolink` для якорей — отвергнуто: новая зависимость ради одной функции; slug считается общей чистой функцией `slugifyHeading` в двух местах (парсер оглавления и рендер h2). (3) Автор в подписи — отвергнуто владельцем. (4) Фабриковать источники ради демо `<Sources>` — отвергнуто: использованы только реальные канонические домены (FTC/Europol).
